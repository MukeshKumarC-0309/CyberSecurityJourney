## Day 09 — Windows CMD & PowerShell

### Why Command Line Over GUI?

In a real pentest you rarely have a graphical interface. When you get a shell on a Windows machine it will be a CMD or PowerShell prompt — no desktop, no File Explorer, no right-click menus. Everything you do will be typed commands.

More importantly, command-line tools can be scripted and automated. A PowerShell one-liner that lists all services running as SYSTEM takes 2 seconds. Clicking through the Services GUI takes 5 minutes.

---

### CMD — Command Prompt

CMD is the legacy Windows shell, inherited from MS-DOS. It's available on every Windows machine without exception. Some old tools and techniques only work in CMD.

#### Navigation
```cmd
cd C:\Users\Admin\Desktop       — change directory (absolute path)
cd ..                           — go up one directory
cd ..\..                        — go up two directories
dir                             — list files and folders
dir /a                          — list including hidden files
dir /s *.txt                    — search recursively for .txt files
cls                             — clear screen
```

#### File Operations
```cmd
type filename.txt               — print file contents
copy source.txt dest.txt        — copy a file
move source.txt C:\dest\        — move a file
del filename.txt                — delete a file
mkdir newfolder                 — create a directory
rmdir /s /q foldername          — delete a folder and all contents
ren oldname.txt newname.txt     — rename a file
```

#### System Information
```cmd
ipconfig                        — basic network configuration
ipconfig /all                   — detailed — shows MAC, DNS, DHCP server
ipconfig /displaydns            — show DNS cache
netstat -an                     — all active connections with ports
netstat -anob                   — connections with owning process ID
systeminfo                      — full system details — OS, patches, hardware
hostname                        — computer name
set                             — list all environment variables
echo %USERNAME%                 — print current username
echo %COMPUTERNAME%             — print machine name
echo %PATH%                     — print the PATH variable
```

#### User and Network Commands
```cmd
net user                        — list local users
net user USERNAME PASSWORD /add — create a new local user
net localgroup Administrators USERNAME /add  — add user to admin group
net share                       — list shared folders
net use                         — list mapped network drives
arp -a                          — show ARP cache
route print                     — show routing table
```

#### Finding Files — Critical for Post-Exploitation
```cmd
dir /s /b C:\*.txt              — find all .txt files from C:\
dir /s /b C:\ | findstr password — find files with "password" in name
findstr /si "password" *.txt    — search file CONTENTS for "password"
findstr /si "password" *.xml *.ini *.config  — common config file types
```

**These search commands are real post-exploitation techniques** — after getting a shell, searching for files containing passwords or credentials is one of the first things you do.

---

### PowerShell

PowerShell is the modern Windows shell. It's built on .NET and works with objects rather than text — this makes it far more powerful than CMD for automation and scripting.

#### PowerShell Basics
```powershell
Get-Help Get-Process            — help for any cmdlet
Get-Help Get-Process -Examples  — examples of cmdlet usage
Get-Command *service*           — find cmdlets related to services
Get-Alias ls                    — what does 'ls' map to? (Get-ChildItem)
```

#### File System
```powershell
Get-ChildItem                   — list files (alias: ls, dir)
Get-ChildItem -Hidden           — include hidden files
Get-ChildItem -Recurse *.txt    — recursive search for .txt files
Get-Content file.txt            — read file (alias: cat, type)
Set-Content file.txt "text"     — write to file
Add-Content file.txt "text"     — append to file
Copy-Item src dest              — copy file
Remove-Item file.txt            — delete file
```

#### System Information
```powershell
Get-ComputerInfo                — comprehensive system info
Get-Process                     — all running processes
Get-Process | Sort-Object CPU -Descending | Select-Object -First 10
Get-Service                     — all services
Get-Service | Where-Object {$_.Status -eq "Running"}
Get-NetIPAddress                — IP configuration
Get-NetTCPConnection            — active TCP connections
Get-NetTCPConnection | Where-Object {$_.State -eq "Listen"}   — listening ports only
```

#### User and Group Management
```powershell
Get-LocalUser                   — list local users
Get-LocalUser | Select-Object Name, Enabled, LastLogon
Get-LocalGroup                  — list local groups
Get-LocalGroupMember Administrators
New-LocalUser -Name "testuser" -Password (ConvertTo-SecureString "Pass123!" -AsPlainText -Force)
Add-LocalGroupMember -Group Administrators -Member testuser
```

#### The Pipeline — PowerShell's Superpower
```powershell
Get-Process | Where-Object {$_.CPU -gt 100}     — filter by CPU
Get-Service | Select-Object Name, Status        — select specific properties
Get-Process | Sort-Object WorkingSet -Descending | Select-Object -First 5
Get-LocalUser | Where-Object {$_.Enabled -eq $true} | Select-Object Name
```

The pipe `|` passes objects (not just text) from one cmdlet to the next. `Where-Object` filters, `Select-Object` picks columns, `Sort-Object` sorts.

#### Finding Sensitive Information — Post-Exploitation
```powershell
# Search file contents recursively
Get-ChildItem -Recurse C:\ -ErrorAction SilentlyContinue | Select-String -Pattern "password"

# Find recently modified files
Get-ChildItem -Recurse C:\Users -ErrorAction SilentlyContinue | Where-Object {$_.LastWriteTime -gt (Get-Date).AddDays(-7)}

# Find all services running as a specific account
Get-WmiObject Win32_Service | Where-Object {$_.StartName -like "*SYSTEM*"} | Select-Object Name, StartName, PathName
```

---

### CMD vs PowerShell — When to Use Which

| Scenario | Use |
|---|---|
| Quick system info | Either — CMD is faster to type |
| Scripting and automation | PowerShell — objects > text |
| Old tools and exploits | CMD — better compatibility |
| Finding and filtering data | PowerShell — pipelines are powerful |
| Running .bat files | CMD |
| Running .ps1 scripts | PowerShell |

## Day 10 — VirtualBox Lab Setup

### Why Virtual Machines for Security Testing?

A VM is an isolated computer running inside your real computer. For pentesting:
- **Isolation** — if malware escapes a VM it's (usually) contained
- **Snapshots** — instantly revert to a clean state after breaking something
- **Multiple OS** — run Linux, old Windows versions, vulnerable targets simultaneously
- **Legal targets** — VMs give you machines you're allowed to attack

---

### VirtualBox Concepts

**Host machine** — your real computer (Windows with 32GB RAM, Core Ultra 7)  
**Guest machine** — the VM running inside VirtualBox  
**Snapshot** — a saved state of a VM — revert to it anytime  
**OVA file** — a portable VM package that VirtualBox can import directly  

---

### Network Modes — Critical for Lab Setup

VirtualBox has several network modes. Choosing the wrong one is a common mistake.

| Mode | VM can reach | Host can reach VM | VMs reach each other | Internet |
|---|---|---|---|---|
| NAT | Internet | No (by default) | No | Yes |
| Bridged | Everything | Yes | Yes | Yes |
| Host-Only | Host only | Yes | Yes | No |
| Internal | Other VMs | No | Yes | No |

**For your lab:**
- **Host-Only** for target VMs (Metasploitable etc.) — isolated, safe, no internet
- **NAT** for your Kali VM when you need to download tools
- **Bridged** only when you need the VM on your real network

---

### Snapshot Strategy

Snapshots are your safety net. Take them:
1. **After initial OS install** — clean baseline
2. **After installing tools** — configured but untouched
3. **Before attempting anything destructive** — you can always revert

```
Right-click VM → Snapshots → Take Snapshot → name it descriptively
e.g. "Clean Ubuntu 22.04 — tools installed — no modifications"
```

To revert: Right-click snapshot → Restore Snapshot

---

### Recommended VM Allocation (Your Machine)

With 32GB RAM and 8 cores you can run:

| VM | RAM | Cores | Storage | Purpose |
|---|---|---|---|---|
| Kali Linux | 6GB | 3 | 60GB | Attacker machine |
| Ubuntu 22.04 | 4GB | 2 | 40GB | Linux learning (Phase 2) |
| Metasploitable 2 | 1GB | 1 | 10GB | Vulnerable target (Phase 3) |
| Windows 10 Eval | 4GB | 2 | 60GB | Windows target / learning |

Total: 15GB RAM, 6 cores — well within your 32GB/8 core limit. Your host Windows keeps the remaining 17GB and 2 cores.

---

### VirtualBox Guest Additions

After installing any VM, install Guest Additions (Devices menu inside the running VM). This enables:
- Proper screen resolution scaling
- Shared clipboard between host and guest
- Drag-and-drop file transfer
- Shared folders

## Day 11 — CIA Triad, Threat Modeling & Core Vocabulary

### The CIA Triad

The CIA Triad is the foundational framework of information security. Every security control, every policy, and every attack can be mapped to one or more of these three properties.

```
        Confidentiality
           /      \
          /        \
    Integrity ---- Availability
```

#### Confidentiality
**Definition:** Information is only accessible to those authorised to see it.

**What breaks it:**
- Unauthorised access (hacking, stolen credentials)
- Unencrypted data in transit (HTTP instead of HTTPS)
- Misconfigured permissions (S3 buckets set to public)
- Social engineering (phishing for credentials)
- Physical theft of unencrypted storage

**Real-world examples:**
- 2017 Equifax breach — 147 million people's personal data exposed
- Any credential dump — usernames and passwords made public
- Wiretapping unencrypted communications

**Defences:** Encryption at rest and in transit, strong access controls, least privilege, MFA

---

#### Integrity
**Definition:** Data is accurate and has not been tampered with by unauthorised parties.

**What breaks it:**
- Man-in-the-middle attacks modifying data in transit
- SQL injection modifying database records
- Malware corrupting or modifying files
- Insider threats altering financial records
- Supply chain attacks (modifying software before distribution)

**Real-world examples:**
- 2020 SolarWinds attack — malicious code inserted into legitimate software updates
- DNS poisoning — altering DNS records to redirect traffic
- BGP hijacking — modifying routing tables to redirect internet traffic

**Defences:** Cryptographic hashing, digital signatures, checksums, audit logs, file integrity monitoring

---

#### Availability
**Definition:** Systems and data are accessible when authorised users need them.

**What breaks it:**
- DDoS attacks (Distributed Denial of Service) — flood a server with traffic
- Ransomware — encrypts files, makes them inaccessible
- Hardware failure without backups
- Natural disasters, power outages
- Misconfigured firewalls blocking legitimate traffic

**Real-world examples:**
- 2016 Mirai botnet DDoS — took down major internet services for hours
- Any ransomware attack — WannaCry, REvil, LockBit

**Defences:** Redundancy, backups, DDoS mitigation, business continuity planning

---

### Core Vocabulary

Knowing these terms precisely matters — in a job interview, a CTF write-up, or a pentest report, using them correctly signals real understanding.

#### Vulnerability vs Exploit vs Payload

| Term | Definition | Example |
|---|---|---|
| **Vulnerability** | A weakness in a system that could be exploited | SQL injection flaw in login form |
| **Exploit** | Code or technique that takes advantage of a vulnerability | The specific SQL string that bypasses login |
| **Payload** | What gets executed after exploitation succeeds | A reverse shell that connects back to the attacker |
| **Attack vector** | The path used to reach and exploit the vulnerability | The login form via HTTP |
| **Attack surface** | All points where an attacker could interact with a target | Every open port, web form, API endpoint |

#### Other Key Terms

| Term | Definition |
|---|---|
| **Zero-day (0-day)** | A vulnerability with no patch available — vendor doesn't know or hasn't fixed it |
| **CVE** | Common Vulnerabilities and Exposures — standardised identifier for known vulnerabilities (e.g. CVE-2021-44228) |
| **CVSS** | Common Vulnerability Scoring System — numeric score 0–10 for severity |
| **Threat actor** | The person or group conducting an attack |
| **Threat** | A potential danger — a threat actor + a vulnerability + motivation |
| **Risk** | Likelihood × Impact — how much a threat actually matters |
| **Patch** | A software update that fixes a vulnerability |
| **Proof of Concept (PoC)** | Code demonstrating that a vulnerability is exploitable |
| **Reverse shell** | Shell that connects FROM the target TO the attacker |
| **Bind shell** | Shell that opens a port ON the target — attacker connects to it |
| **Lateral movement** | Moving from one compromised machine to others in the same network |
| **Persistence** | Mechanisms that ensure attacker access survives reboots |
| **Exfiltration** | Stealing data out of a compromised network |
| **C2 / C&C** | Command and Control — attacker's infrastructure that malware reports back to |
| **Foothold** | Initial, usually low-privilege access to a target |
| **Privilege escalation** | Going from low privilege to high privilege (e.g. user → root/SYSTEM) |

---

### Threat Modeling

Threat modeling is the process of systematically identifying what can go wrong. The basic questions:

1. **What are we protecting?** (assets)
2. **Who might attack us?** (threat actors)
3. **How might they attack?** (attack vectors)
4. **What's the impact if they succeed?** (risk)
5. **How do we mitigate?** (controls)

#### Threat Actor Types

| Type | Motivation | Sophistication | Examples |
|---|---|---|---|
| Script kiddie | Fame, fun | Low — uses existing tools | Defacing websites |
| Hacktivist | Political/social cause | Medium | Anonymous |
| Cybercriminal | Financial gain | Medium-High | Ransomware gangs |
| Insider threat | Revenge, money | Varies | Disgruntled employee |
| Nation-state | Espionage, disruption | Very High | APT groups |

---

### Real Breaches Mapped to CIA Triad

**Equifax 2017** — Confidentiality failure
- Apache Struts vulnerability (CVE-2017-5638) exploited
- 147 million SSNs, birth dates, addresses stolen
- Patch was available 2 months before breach — not applied

**SolarWinds 2020** — Integrity failure
- Attackers inserted malicious code into SolarWinds Orion software updates
- 18,000 organisations installed the trojanised update
- Including US government agencies

**WannaCry 2017** — Availability failure
- Ransomware spread via EternalBlue (NSA exploit leaked by Shadow Brokers)
- Encrypted files on 230,000+ computers in 150 countries
- NHS hospitals in UK had to cancel surgeries

## Day 12 — Windows Registry

### What is the Windows Registry?

The Registry is a hierarchical database that stores configuration settings for Windows, hardware, installed software, and user preferences. It is Windows' central nervous system — almost everything the OS does involves reading from or writing to the registry.

As a pentester, the registry matters for three reasons:
1. **Persistence** — malware adds itself to Run keys to survive reboots
2. **Credential storage** — some software stores passwords in registry keys
3. **Privilege escalation** — misconfigured registry key permissions allow overwriting service executable paths

---

### The 5 Root Hives

```
HKEY_LOCAL_MACHINE (HKLM)       — system-wide settings, all users
HKEY_CURRENT_USER (HKCU)        — settings for the logged-in user only
HKEY_CLASSES_ROOT (HKCR)        — file type associations and COM objects
HKEY_USERS (HKU)                — settings for ALL user profiles
HKEY_CURRENT_CONFIG (HKCC)      — current hardware profile
```

**HKLM vs HKCU distinction:**
- HKLM changes affect everyone on the machine — requires Administrator
- HKCU changes affect only the current user — writable by standard users
- This distinction matters for persistence: HKCU Run key doesn't require admin, which is why malware often uses it

---

### Registry Data Types

| Type | Name | Description | Example |
|---|---|---|---|
| REG_SZ | String | Plain text | `C:\Windows\System32\notepad.exe` |
| REG_EXPAND_SZ | Expandable String | String with variables | `%SystemRoot%\notepad.exe` |
| REG_DWORD | 32-bit integer | Numeric values | `1` or `0` for enabled/disabled |
| REG_QWORD | 64-bit integer | Large numbers | Timestamps |
| REG_BINARY | Binary | Raw binary data | Hardware config |
| REG_MULTI_SZ | Multi-string | Array of strings | List of services |

---

### Critical Registry Locations for Pentesting

#### Persistence — Run Keys
Every value in these keys is a program that runs automatically at startup.

```
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce
HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce
```

`RunOnce` — runs once then deletes itself  
`Run` — runs every time  
HKLM version runs before login, HKCU runs after login for that specific user

**How malware uses this:**
```
Key: HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
Value name: WindowsDefenderUpdate
Value data: C:\Users\victim\AppData\Roaming\malware.exe
```

Looks legitimate, runs on every boot, hard to spot without checking these keys.

**To enumerate run keys with PowerShell:**
```powershell
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Run"
Get-ItemProperty "HKCU:\SOFTWARE\Microsoft\Windows\CurrentVersion\Run"
```

---

#### System Information
```
HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion
```
Contains: Windows version, build number, install date, registered owner, product name.

In a pentest: read this to fingerprint the OS precisely, which tells you which exploits and patches to look for.

---

#### Services
```
HKLM\SYSTEM\CurrentControlSet\Services\SERVICENAME
```

Each service has its own key. Key values:
- `ImagePath` — path to the executable (can be a privesc if writable)
- `ObjectName` — account the service runs as (LocalSystem = SYSTEM)
- `Start` — 2=Auto, 3=Manual, 4=Disabled
- `Type` — 16=standalone process, 32=shared process

**Unquoted Service Path vulnerability:**
If `ImagePath` is `C:\Program Files\My App\service.exe` (no quotes), Windows tries to run:
1. `C:\Program.exe` first
2. `C:\Program Files\My.exe` second
3. `C:\Program Files\My App\service.exe` last

If you can write to `C:\Program Files\My.exe`, you get code execution as whatever account the service runs as.

**Find services with unquoted paths:**
```cmd
wmic service get name,displayname,pathname,startmode | findstr /i "auto" | findstr /i /v "c:\windows\\" | findstr /i /v """
```

---

#### Saved Credentials — Common Locations
```
HKLM\SOFTWARE\OpenSSH\Agent\Keys          — SSH keys
HKCU\SOFTWARE\SimonTatham\PuTTY\Sessions  — PuTTY saved sessions (sometimes contain passwords)
HKLM\SYSTEM\CurrentControlSet\Services\SNMP\Parameters\ValidCommunities  — SNMP community strings
```

---

#### SAM Database Location
```
HKLM\SAM                    — encrypted local account hashes
HKLM\SECURITY               — cached domain credentials
HKLM\SYSTEM                 — SYSKEY needed to decrypt SAM
```

These three hives together allow extracting all local password hashes with tools like secretsdump. They require SYSTEM-level access to read.

---

### Navigating the Registry — Tools

#### regedit (GUI)
```
Win+R → regedit → Enter
```
Navigate like a file browser. Use Ctrl+F to search for values.

**Caution:** Deleting the wrong key can break Windows. Never modify anything you don't understand. Always export a key before modifying it: File → Export.

#### reg.exe (CMD)
```cmd
reg query "HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion"
reg query "HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run"
reg query HKLM /f "password" /t REG_SZ /s          — search all HKLM for "password"
reg add "HKCU\...\Run" /v TestEntry /t REG_SZ /d "notepad.exe"  — add a value
reg delete "HKCU\...\Run" /v TestEntry /f           — delete a value
reg export "HKCU\Software" backup.reg               — export a key to file
```

#### PowerShell
```powershell
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion"
Get-ChildItem "HKLM:\SYSTEM\CurrentControlSet\Services"
Get-ItemProperty "HKCU:\SOFTWARE\Microsoft\Windows\CurrentVersion\Run"

# Search registry for passwords
Get-ChildItem -Path HKLM:\ -Recurse -ErrorAction SilentlyContinue |
    Get-ItemProperty -ErrorAction SilentlyContinue |
    Where-Object {$_ -match "password"}
```

---

### The Persistence Demo — What You Did on Day 12

You added a test Run key entry and observed it survive a reboot:

```
Key:   HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
Name:  TestPersistence
Value: notepad.exe
```

After reboot → Notepad opened automatically. That's exactly what malware does.

**Clean up:** Delete `TestPersistence` value from the Run key after testing.

This is one of the most important things to remember going into Phase 3 — after getting a shell, adding a Run key is how you maintain access. The only difference from what you did is the payload is a reverse shell executable instead of notepad.
