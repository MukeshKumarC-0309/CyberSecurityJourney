## Day 05 — Wireshark Deep Dive

### Why Wireshark Matters for Pentesting

Wireshark is a **packet analyser** — it captures every packet that flows through your network interface and lets you inspect it layer by layer. Pentesters use it to:
- Analyse captured network traffic for credentials and sensitive data
- Understand what protocols a target is using
- Debug exploits that aren't working
- Identify unusual traffic patterns that indicate a compromise
- Solve CTF challenges involving packet capture (.pcap) files

---

### Wireshark Interface

```
[Menu Bar]
[Toolbar — start/stop capture, filters]
[Packet List Pane]     ← one row per packet
[Packet Details Pane]  ← layers expanded as a tree
[Packet Bytes Pane]    ← raw hex and ASCII
```

The **Packet Details Pane** is where the OSI model becomes real — you'll see each layer's header expanded and labelled. A typical HTTP packet shows:
```
▼ Frame 1 (Physical/Data Link metadata)
▼ Ethernet II (Layer 2 — src/dst MAC addresses)
▼ Internet Protocol Version 4 (Layer 3 — src/dst IP)
▼ Transmission Control Protocol (Layer 4 — src/dst ports, flags)
▼ Hypertext Transfer Protocol (Layer 7 — method, host, path)
```

---

### Essential Wireshark Filters

Filters are written in Wireshark's display filter language. Apply them in the green filter bar at the top.

#### Protocol Filters
```
tcp                         — all TCP traffic
udp                         — all UDP traffic
http                        — HTTP only (note: not HTTPS)
dns                         — all DNS queries and responses
arp                         — all ARP traffic
icmp                        — ping traffic
tls                         — TLS handshakes and encrypted data
```

#### IP Address Filters
```
ip.addr == 192.168.1.1      — all traffic to OR from this IP
ip.src == 192.168.1.1       — only traffic FROM this IP
ip.dst == 192.168.1.1       — only traffic TO this IP
ip.addr == 192.168.1.0/24   — entire subnet
```

#### Port Filters
```
tcp.port == 80              — HTTP traffic
tcp.port == 443             — HTTPS traffic
tcp.port == 22              — SSH traffic
tcp.dstport == 80           — only requests going TO port 80
tcp.srcport == 80           — only responses coming FROM port 80
```

#### TCP Flag Filters
```
tcp.flags.syn == 1                          — SYN packets (connection starts)
tcp.flags.syn == 1 && tcp.flags.ack == 0   — only SYN (not SYN-ACK)
tcp.flags.rst == 1                          — RST packets (connection resets)
tcp.flags.fin == 1                          — FIN packets (connection closes)
```

#### Combining Filters
```
http && ip.addr == 93.184.216.34            — HTTP traffic to/from a specific IP
tcp.port == 80 || tcp.port == 443           — HTTP or HTTPS
!(arp || dns || icmp)                       — exclude noisy protocols
http.request.method == "POST"               — only POST requests
http contains "password"                    — HTTP traffic containing the word "password"
```

---

### Following Streams

**Follow TCP Stream:** Right-click any packet → Analyze → Follow → TCP Stream

This reassembles the full bidirectional conversation in readable form. Red = client to server, Blue = server to client.

**Example — what you see in a raw HTTP stream:**
```
GET / HTTP/1.1
Host: neverssl.com
User-Agent: Mozilla/5.0
Accept: text/html

HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 2748

<!DOCTYPE html>
<html>...
```

**Follow UDP Stream:** Same process — useful for DNS queries.

---

### Reading a DNS Query in Wireshark

Filter: `dns`

Click a DNS query packet. In the Packet Details:
```
▼ Domain Name System (query)
    Transaction ID: 0x1234
    Flags: Standard query
    ▼ Queries
        example.com: type A, class IN
```

Click the response packet:
```
▼ Domain Name System (response)
    Transaction ID: 0x1234 (matches the query)
    Flags: Standard response, no error
    ▼ Answers
        example.com: type A, class IN, addr 93.184.216.34
    Time to live: 3600
```

The **Transaction ID** links query to response. In a DNS spoofing attack, the attacker races to send a fake response with the correct Transaction ID before the real server responds.

---

### Credential Capture in Wireshark

On unencrypted HTTP traffic you can extract credentials directly.

Filter: `http.request.method == "POST"`

Find a login POST request. Right-click → Follow TCP Stream. In the stream you'll see something like:
```
POST /login HTTP/1.1
Host: target.com
Content-Type: application/x-www-form-urlencoded

username=admin&password=secretpassword123
```

This is exactly how credentials are extracted from captured traffic in CTFs and real network pentests. **This is why HTTPS matters** — this attack only works on HTTP.

---

### Wireshark and .pcap Files

Many CTF challenges give you a `.pcap` file to analyse offline. Open it with: File → Open → select the .pcap

All the same filters and stream-following techniques work on captured files. Common things to look for in CTF .pcap files:
- Credentials in HTTP POST requests
- Flags in HTTP responses
- Unusual protocols that hint at a challenge
- DNS queries revealing hidden subdomains
- FTP traffic (unencrypted — credentials visible in plaintext)

## Day 06 — Python Sockets

### Why Python Sockets for Pentesting?

Python socket programming lets you create custom network tools from scratch. Understanding sockets means you understand exactly what every network tool does under the hood. Most pentesting scripts — port scanners, banner grabbers, reverse shells, network fuzzers — are built on top of raw sockets.

---

### Socket Fundamentals

A socket is an endpoint for network communication. Think of it as a phone — you can use it to call someone (client) or wait for someone to call you (server).

#### Socket Types
```python
socket.AF_INET      # IPv4 addressing
socket.AF_INET6     # IPv6 addressing
socket.SOCK_STREAM  # TCP — reliable, connection-oriented
socket.SOCK_DGRAM   # UDP — fast, connectionless
```

#### Client vs Server Socket Workflow

```
CLIENT                          SERVER
socket()                        socket()
connect(host, port)             bind(host, port)
    |                           listen()
    |                           accept()  ← blocks here waiting
    |<====== connection =======>|
send() / recv()                 send() / recv()
close()                         close()
```

---

### Script 1 — TCP Client

```python
import socket

target = "neverssl.com"
port = 80

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect((target, port))

request = b"GET / HTTP/1.1\r\nHost: neverssl.com\r\n\r\n"
s.send(request)

response = s.recv(4096)
print(response.decode())
s.close()
```

**Line by line breakdown:**
- `socket.socket(AF_INET, SOCK_STREAM)` — create a TCP socket
- `connect((target, port))` — note the double parentheses — takes a tuple
- `b"..."` — the `b` prefix makes it a bytes object — sockets send/receive bytes, not strings
- `\r\n` — HTTP requires carriage return + newline to separate headers
- `\r\n\r\n` — double blank line signals end of HTTP headers
- `recv(4096)` — receive up to 4096 bytes — call multiple times for large responses
- `decode()` — convert bytes back to a readable string

---

### Script 2 — TCP Server

```python
import socket

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.bind(("0.0.0.0", 9999))
server.listen(5)
print("[*] Listening on port 9999...")

client, addr = server.accept()
print(f"[+] Connection from {addr}")

client.send(b"Hello from your Python server!\n")
client.close()
```

**Key concepts:**
- `bind(("0.0.0.0", 9999))` — `0.0.0.0` means listen on all network interfaces
- `listen(5)` — allow up to 5 queued connections waiting to be accepted
- `accept()` — blocks (pauses) until a client connects, returns (socket, address) tuple
- This is **exactly** how a netcat listener works — same principle, different tool

**To test:** Run the server in one terminal, then connect from another:
```cmd
telnet 127.0.0.1 9999
```

---

### Script 3 — Banner Grabber

Banner grabbing is a recon technique — you connect to a port and read what the service announces about itself. Many services immediately send a banner identifying their software and version.

```python
import socket

def grab_banner(ip, port):
    try:
        s = socket.socket()
        s.settimeout(2)
        s.connect((ip, port))
        s.send(b"HEAD / HTTP/1.0\r\n\r\n")
        banner = s.recv(1024).decode().strip()
        print(f"[+] Port {port}: {banner[:100]}")
        s.close()
    except:
        print(f"[-] Port {port}: No banner or closed")

target = "neverssl.com"
ports = [80, 443, 21, 22, 25]

for port in ports:
    grab_banner(target, port)
```

**What you learn from banners:**
```
Port 22: SSH-2.0-OpenSSH_8.2p1 Ubuntu-4ubuntu0.5
          ^version  ^software   ^OS hint
```

This is the same information Nmap's `-sV` flag extracts — you just built a primitive version of it.

---

### Script 4 — Port Scanner

```python
import socket

def scan_port(ip, port):
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.settimeout(1)
    result = s.connect_ex((ip, port))
    s.close()
    return result == 0

target = "127.0.0.1"

print(f"[*] Scanning {target}")
for port in range(1, 1025):
    if scan_port(target, port):
        print(f"[+] Port {port} OPEN")
```

**Key difference:** `connect_ex()` vs `connect()`
- `connect()` raises an exception if connection fails
- `connect_ex()` returns an error code — 0 means success (port open), non-zero means failure

This is the core logic of every port scanner. Nmap does the same thing but with much more sophistication — stealth techniques, service detection, OS fingerprinting.

## Day 08 — Windows Internals: Users, Groups & Permissions

### Why Windows Internals for Pentesting?

Most corporate environments run Windows. Understanding how Windows manages users, permissions, and processes is essential because:
- **Privilege escalation** requires knowing what you have and what you need
- **Lateral movement** means understanding user relationships across machines
- **Persistence** mechanisms rely on Windows-specific registry keys and services
- **Post-exploitation** commands are all Windows-native tools

---

### Windows Account Types

#### Local Accounts
Exist only on the machine. Stored in the local Security Account Manager (SAM) database at `C:\Windows\System32\config\SAM` — encrypted and inaccessible while Windows is running.

#### Domain Accounts
Managed by Active Directory (AD) on a domain controller. Used in corporate environments — one account, login from any domain-joined machine.

#### Built-in Accounts

| Account | Description | Privilege Level |
|---|---|---|
| Administrator | Built-in local admin — often disabled by default on Win10+ | Very High |
| SYSTEM | The OS itself — highest privilege, above Administrator | Highest |
| Local Service | Used by services needing minimal permissions | Low |
| Network Service | Services that need network access | Low-Medium |
| Guest | Restricted, usually disabled | Very Low |

**SYSTEM vs Administrator:**
- Administrator can be locked out, have its password changed, be disabled
- SYSTEM cannot be logged into interactively — it IS the OS
- SYSTEM has access to everything including memory of all processes
- Getting SYSTEM access = complete control of the machine
- This is the ultimate goal of Windows privilege escalation

---

### Security Identifiers (SIDs)

Every account and group has a unique SID. Format: `S-1-5-21-XXXXXXXXXX-XXXXXXXXXX-XXXXXXXXXX-RID`

**Well-known SIDs:**
| SID | Account |
|---|---|
| S-1-5-18 | SYSTEM |
| S-1-5-19 | Local Service |
| S-1-5-20 | Network Service |
| S-1-1-0 | Everyone |
| S-1-5-32-544 | Administrators group |

```powershell
whoami /user        # see your own SID
```

---

### Windows Privileges

Privileges are special rights assigned to accounts. Different from permissions (which control file access), privileges control what the account can do at the system level.

```cmd
whoami /priv        # list all privileges for current account
```

**High-value privileges for privilege escalation:**

| Privilege | What it allows | Exploit potential |
|---|---|---|
| SeImpersonatePrivilege | Impersonate another user's token | Potato attacks → SYSTEM |
| SeDebugPrivilege | Debug any process including SYSTEM processes | Read/inject into SYSTEM processes |
| SeBackupPrivilege | Read any file regardless of ACL | Read SAM database → extract hashes |
| SeRestorePrivilege | Write any file regardless of ACL | Overwrite system files |
| SeTakeOwnershipPrivilege | Take ownership of any object | Own any file or registry key |

**If you ever see SeImpersonatePrivilege enabled on a compromised account in Phase 3 — you have a fast path to SYSTEM.** Look up "Juicy Potato" or "PrintSpoofer" attacks.

---

### Access Control Lists (ACLs)

Windows uses ACLs to control who can access files, folders, registry keys, and other objects.

#### ACL Structure
```
Object (file/folder/key)
└── DACL (Discretionary ACL) — who can access and how
    ├── ACE (Access Control Entry) — ALLOW SYSTEM Full Control
    ├── ACE — ALLOW Administrators Full Control
    ├── ACE — ALLOW Users Read & Execute
    └── ACE — DENY Guest All
```

#### NTFS Permissions
| Permission | File | Folder |
|---|---|---|
| Full Control | Read, write, execute, delete, change permissions | All file permissions + create/delete files |
| Modify | Read, write, execute, delete | Read, write, execute + create/delete |
| Read & Execute | Read + run | Browse + run files inside |
| Read | View content | View files listed |
| Write | Modify content | Create new files |

**From PowerShell:**
```powershell
Get-Acl "C:\path\to\file" | Format-List
```

**From CMD:**
```cmd
icacls "C:\path\to\file"
```

`icacls` output legend:
- `(F)` — Full Control
- `(M)` — Modify
- `(RX)` — Read & Execute
- `(R)` — Read
- `(W)` — Write
- `(I)` — Inherited from parent

---

### Key Commands for User Enumeration

These are your first commands on any Windows machine during a pentest — run them immediately after getting access to understand the account you're working with.

```cmd
whoami                          — current username
whoami /priv                    — current account's privileges
whoami /groups                  — groups the current account belongs to
whoami /all                     — everything above at once

net user                        — list all local user accounts
net user USERNAME               — details about a specific user
net localgroup                  — list all local groups
net localgroup Administrators   — list members of the Administrators group
net localgroup "Remote Desktop Users"   — who can RDP?
```

```powershell
Get-LocalUser                   — list local users (PowerShell)
Get-LocalGroup                  — list local groups
Get-LocalGroupMember Administrators     — members of Administrators
```

---

### Exploring Processes and Services

```cmd
tasklist                        — list all running processes
tasklist /svc                   — map processes to their services
tasklist /fi "USERNAME eq SYSTEM"       — processes running as SYSTEM
```

```powershell
Get-Process | Sort-Object CPU -Descending | Select-Object -First 20
Get-Service | Where-Object {$_.Status -eq "Running"}
Get-Service | Where-Object {$_.StartType -eq "Automatic"}
Get-WmiObject Win32_Service | Where-Object {$_.StartName -eq "LocalSystem"} | Select-Object Name, PathName
```

The last command is gold — it lists every service running as LocalSystem (SYSTEM). In Phase 3 you'll look for misconfigured services in this list as privesc paths.
