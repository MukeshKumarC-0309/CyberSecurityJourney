### Python Port Scanner — Full Version

Building on Day 6's basic scanner, today's version adds banner grabbing, threading for speed, and proper output formatting.

```python
import socket
import threading
from datetime import datetime

target = "127.0.0.1"
open_ports = []
lock = threading.Lock()

def scan_port(port):
    try:
        s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        s.settimeout(1)
        result = s.connect_ex((target, port))
        if result == 0:
            try:
                s.send(b"HEAD / HTTP/1.0\r\n\r\n")
                banner = s.recv(1024).decode().strip()[:50]
            except:
                banner = "No banner"
            with lock:
                open_ports.append((port, banner))
                print(f"[+] Port {port} OPEN | {banner}")
        s.close()
    except:
        pass

print(f"[*] Scanning {target}")
print(f"[*] Started: {datetime.now()}")
print("-" * 50)

threads = []
for port in range(1, 1025):
    t = threading.Thread(target=scan_port, args=(port,))
    threads.append(t)
    t.start()

for t in threads:
    t.join()

print("-" * 50)
print(f"[*] Scan complete. {len(open_ports)} open ports found.")
print(f"[*] Finished: {datetime.now()}")
```

**New concepts vs Day 6:**
- `threading.Thread` — runs each port scan simultaneously instead of one at a time — dramatically faster
- `lock` — a threading lock prevents two threads writing to `open_ports` at the same time (race condition)
- `t.join()` — waits for all threads to finish before printing the summary
- Banner grabbing integrated — shows service info alongside port number

## Day 14 — Cryptography Basics

### Why Cryptography for Pentesting

Cryptography is everywhere in security — passwords are hashed, communications are encrypted, tokens are signed. Understanding the fundamentals means:
- You know what's crackable and what isn't
- You recognise weak implementations that can be exploited
- You can work with CTF crypto challenges
- You understand what tools like John the Ripper and hashcat are actually doing

---

### Hashing vs Encryption — Critical Distinction

| Property | Hashing | Encryption |
|---|---|---|
| Reversible? | No — one-way only | Yes — decrypt with key |
| Purpose | Verify integrity/store passwords | Protect data in transit/at rest |
| Output | Fixed length regardless of input | Variable length |
| Key required? | No | Yes |
| Example use | Storing passwords | HTTPS, file encryption |

**The most common beginner mistake:** saying passwords are "encrypted." They're not — they're **hashed**. Encryption is reversible. Hashing is not (without brute force).

---

### Common Hashing Algorithms

| Algorithm | Output Length | Status | Notes |
|---|---|---|---|
| MD5 | 128 bits (32 hex chars) | Broken — do not use | Still seen everywhere, fast to crack |
| SHA-1 | 160 bits (40 hex chars) | Weak — deprecated | Collision attacks demonstrated |
| SHA-256 | 256 bits (64 hex chars) | Strong | Part of SHA-2 family, widely used |
| SHA-512 | 512 bits (128 hex chars) | Strong | Slower, used where extra security needed |
| bcrypt | Variable | Strong | Designed for passwords — slow by design |
| Argon2 | Variable | Very Strong | Modern password hashing standard |

**Why bcrypt is better for passwords than SHA-256:**
SHA-256 is fast — that's good for integrity checks but bad for passwords. Fast hashing means an attacker can try billions of passwords per second. bcrypt is intentionally slow (configurable work factor) — the same attack takes years instead of hours.

---

### Hashing in Python

```python
import hashlib

text = "password123"

# MD5
md5 = hashlib.md5(text.encode()).hexdigest()
print(f"MD5:    {md5}")
# Output: 482c811da5d5b4bc6d497ffa98491e38

# SHA-1
sha1 = hashlib.sha1(text.encode()).hexdigest()
print(f"SHA-1:  {sha1}")

# SHA-256
sha256 = hashlib.sha256(text.encode()).hexdigest()
print(f"SHA-256: {sha256}")

# SHA-512
sha512 = hashlib.sha512(text.encode()).hexdigest()
print(f"SHA-512: {sha512}")

# Observe: all produce fixed length regardless of input length
text2 = "a"
print(f"\nSHA-256 of 'a':            {hashlib.sha256(text2.encode()).hexdigest()}")
print(f"SHA-256 of 'password123':  {hashlib.sha256(text.encode()).hexdigest()}")
# Both outputs are exactly 64 characters
```

---

### Salting — Why It Matters

**The problem with plain hashing:**
If two users have the same password, their hashes are identical. An attacker who gets the database can immediately see which accounts share passwords. Worse — precomputed **rainbow tables** (giant lookup tables of hash→password) make cracking trivial.

**The solution — salting:**
A salt is a random unique string added to each password before hashing.

```
password123 + "xK9#mQ2p" (salt) → hash this combined string
```

Now two users with the same password have completely different hashes. Rainbow tables are useless because the table would need to be recomputed for every possible salt.

```python
import hashlib
import os

password = "password123"

# Generate a random salt
salt = os.urandom(32)

# Hash password + salt together
salted_hash = hashlib.pbkdf2_hmac('sha256', password.encode(), salt, 100000)

print(f"Salt: {salt.hex()}")
print(f"Hash: {salted_hash.hex()}")
```

`pbkdf2_hmac` — runs the hash function 100,000 times (the last argument). This makes brute-forcing 100,000x slower. This is the principle behind bcrypt and Argon2.

---

### Base64 Encoding

Base64 is **encoding, not encryption** — it's completely reversible without a key. It converts binary data to ASCII text so it can be safely transmitted in text-based protocols (HTTP headers, email, JSON).

You'll see Base64 constantly in CTFs and web pentesting.

```python
import base64

text = "Hello, Hacker!"

# Encode
encoded = base64.b64encode(text.encode()).decode()
print(f"Encoded: {encoded}")
# Output: SGVsbG8sIEhhY2tlciE=

# Decode
decoded = base64.b64decode(encoded).decode()
print(f"Decoded: {decoded}")
# Output: Hello, Hacker!
```

**Recognition:** Base64 strings end with `=` or `==` padding and use characters A-Z, a-z, 0-9, +, /

**Common CTF scenario:** You find a string like `SGVsbG8=` in a web response or cookie. Decode it — it might contain a flag, a hidden parameter, or credentials.

---

### Identifying Hash Types

In CTFs you're often given a hash and need to identify what type it is before cracking it.

| Hash | Length | Starts with | Type |
|---|---|---|---|
| `482c811da5d5b4bc6d497ffa98491e38` | 32 chars | anything | MD5 |
| `5baa61e4c9b93f3f0682250b6cf8331b7ee68fd8` | 40 chars | anything | SHA-1 |
| `5e884898da28047151d0e56f8dc629...` | 64 chars | anything | SHA-256 |
| `$2b$12$...` | 60 chars | $2b$ or $2y$ | bcrypt |
| `$6$salt$hash` | variable | $6$ | SHA-512crypt (Linux) |
| `$1$salt$hash` | variable | $1$ | MD5crypt (Linux) |

**Linux password hashes** are stored in `/etc/shadow` — the format tells you the algorithm:
- `$1$` = MD5
- `$5$` = SHA-256
- `$6$` = SHA-512 (most common on modern Linux)
- `$y$` = yescrypt (newer systems)

### XOR — The Simplest Cipher

XOR (exclusive OR) is used in many encryption algorithms and CTF challenges. Understanding it is fundamental.

**XOR truth table:**
```
0 XOR 0 = 0
0 XOR 1 = 1
1 XOR 0 = 1
1 XOR 1 = 0
```

Key property: `A XOR B XOR B = A` — XORing with the same key twice returns the original.

```python
# Simple XOR cipher
def xor_cipher(text, key):
    return ''.join(chr(ord(c) ^ ord(key[i % len(key)])) for i, c in enumerate(text))

plaintext = "Hello"
key = "K"
encrypted = xor_cipher(plaintext, key)
decrypted = xor_cipher(encrypted, key)   # XOR again with same key = original

print(f"Encrypted: {[ord(c) for c in encrypted]}")
print(f"Decrypted: {decrypted}")
```

## Day 16 — Linux Intro: Filesystem & Basic Commands

### Why Linux for Pentesting

Kali Linux is Debian-based. Every major pentesting tool — Nmap, Metasploit, Burp Suite, sqlmap, Wireshark — runs natively on Linux. Understanding Linux isn't optional for a pentester — it's foundational. You'll spend more time in Linux terminals during Phase 3 than anywhere else.

Starting with Ubuntu (not Kali) means you learn the OS itself before being handed every tool at once.

---

### The Linux Filesystem Hierarchy

Unlike Windows which organises by drive letter (C:\, D:\), Linux uses a single unified tree starting from `/` (root).

```
/                   — root of everything
├── bin/            — essential binaries (ls, cp, mv, cat)
├── sbin/           — system binaries (only for root — ifconfig, iptables)
├── etc/            — configuration files (ssh config, passwd, hosts)
├── home/           — user home directories (/home/cyberuser)
├── root/           — root user's home directory
├── var/            — variable data — logs (/var/log), web files (/var/www)
├── tmp/            — temporary files — world-writable, deleted on reboot
├── usr/            — user programs and utilities
│   ├── bin/        — non-essential user binaries
│   └── share/      — shared data (Kali: /usr/share/wordlists, /usr/share/nmap)
├── lib/            — shared libraries
├── proc/           — virtual filesystem — running processes and system info
├── dev/            — device files (disks, terminals, /dev/null)
├── mnt/            — mount point for temporary mounts
└── opt/            — optional software installed manually
```

**Key locations for pentesting:**
| Path | Why it matters |
|---|---|
| `/etc/passwd` | List of all user accounts |
| `/etc/shadow` | Hashed passwords (requires root to read) |
| `/etc/hosts` | Local DNS overrides — check for internal hostnames |
| `/etc/crontab` | Scheduled tasks — common privesc location |
| `/etc/ssh/sshd_config` | SSH server configuration |
| `/var/log/` | System logs — auth.log shows login attempts |
| `/tmp/` | World-writable — where attackers upload tools |
| `/home/username/.ssh/` | SSH keys — check for id_rsa files |
| `/usr/share/wordlists/` | Kali — rockyou.txt and other wordlists live here |


### Hidden Files and Directories

In Linux, files and directories starting with `.` are hidden — `ls` won't show them by default.

```bash
ls -la ~                    — show all hidden files in home directory
```

Common hidden files to look for during pentesting:
```
.bash_history               — command history — gold mine for credentials
.ssh/                       — SSH keys
.bashrc / .bash_profile     — shell config — check for aliases and env vars
.gitconfig                  — git credentials
.aws/credentials            — AWS access keys
.gnupg/                     — GPG keys
```

```bash
cat ~/.bash_history          — read command history
```

This is one of the first things you read on a compromised Linux machine — `.bash_history` often contains passwords typed as command arguments, SSH connection details, and database connection strings.
