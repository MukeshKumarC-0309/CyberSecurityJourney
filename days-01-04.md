## Day 01 — The OSI Model

### What is the OSI Model?

The OSI (Open Systems Interconnection) model is a conceptual framework that standardises how different network systems communicate with each other. It was created so that hardware and software from different vendors can work together — your browser made by one company can talk to a server built by another because they both follow the same layered rules.

There are **7 layers**. Data travels **down** the layers on the sender's side (encapsulation) and **up** the layers on the receiver's side (decapsulation).

---

### The 7 Layers — Bottom to Top

#### Layer 1 — Physical
- **What it does:** Transmits raw bits (0s and 1s) over a physical medium
- **Real-world examples:** Ethernet cables, fibre optic cables, Wi-Fi radio signals, USB
- **Devices:** Hubs, repeaters, network cables
- **Attacker relevance:** Physical tapping, jamming Wi-Fi signals, rogue access points

#### Layer 2 — Data Link
- **What it does:** Takes raw bits from Layer 1 and organises them into **frames**. Handles error detection and MAC addressing.
- **Real-world examples:** Ethernet frames, Wi-Fi frames, switches
- **Key concept:** MAC addresses live here — every network interface card has a unique MAC address burned in
- **Protocols:** Ethernet, Wi-Fi (802.11), ARP
- **Attacker relevance:** ARP spoofing, MAC flooding, VLAN hopping

#### Layer 3 — Network
- **What it does:** Handles logical addressing (IP addresses) and routing — deciding which path data takes across multiple networks
- **Real-world examples:** Your router deciding whether to send a packet left or right
- **Key concept:** IP addresses live here. Layer 2 gets data across one network segment; Layer 3 gets data across multiple networks (the internet)
- **Protocols:** IP, ICMP, routing protocols (OSPF, BGP)
- **Attacker relevance:** IP spoofing, ICMP floods (ping of death), route poisoning

#### Layer 4 — Transport
- **What it does:** End-to-end communication between applications. Manages segmentation, flow control, and error recovery.
- **Real-world examples:** A large file being split into smaller chunks for transmission and reassembled at the other end
- **Key concepts:**
  - **TCP** — reliable, connection-oriented, guarantees delivery (used for web, email, SSH)
  - **UDP** — fast, connectionless, no guarantee (used for video streaming, DNS, VoIP)
  - **Ports** — Layer 4 uses port numbers to direct traffic to the right application
- **Attacker relevance:** Port scanning, SYN flood attacks, session hijacking

#### Layer 5 — Session
- **What it does:** Manages sessions (conversations) between applications — opening, maintaining, and closing connections
- **Real-world examples:** When you log into a website, the session layer manages keeping you logged in across multiple requests
- **Protocols:** NetBIOS, RPC, SMB session management
- **Attacker relevance:** Session fixation, session hijacking

#### Layer 6 — Presentation
- **What it does:** Translates data between the application and the network — handles encryption, encoding, and compression
- **Real-world examples:** TLS/SSL encryption lives here, JPEG compression, ASCII to binary conversion
- **Key concept:** This is where HTTPS encryption happens — data is encrypted before going down to Layer 5
- **Attacker relevance:** SSL stripping attacks, downgrade attacks

#### Layer 7 — Application
- **What it does:** The layer closest to the user — provides network services directly to applications
- **Real-world examples:** Your browser sending an HTTP request, your email client sending SMTP
- **Protocols:** HTTP, HTTPS, FTP, SSH, DNS, SMTP, POP3
- **Attacker relevance:** SQL injection, XSS, CSRF, phishing — almost all web attacks happen here

---

### How to Remember the Layers

**Top to bottom:** All People Seem To Need Data Processing  
**Bottom to top:** Please Do Not Throw Sausage Pizza Away

---

### Encapsulation vs Decapsulation

When you send data:
1. Your application creates data (Layer 7)
2. Each layer **adds a header** as it passes down — this is called **encapsulation**
3. By the time it reaches Layer 1 it's raw bits on a wire

When the receiver gets data:
1. Each layer **strips its header** as data passes up — this is **decapsulation**
2. By the time it reaches Layer 7 it's the original data your application sent

This is why a pentester can read a packet at different layers — each layer's header tells you something different about the communication.

---

### Why This Matters for Pentesting

Every attack targets a specific layer:
- **ARP spoofing** → Layer 2
- **IP spoofing** → Layer 3
- **Port scanning** → Layer 4
- **SSL stripping** → Layer 6
- **SQL injection** → Layer 7

When you read about an attack, ask yourself: which layer is being abused? The answer is always in the OSI model.

---

### Key Terms
| Term | Definition |
|---|---|
| Frame | Layer 2 data unit — includes MAC addresses |
| Packet | Layer 3 data unit — includes IP addresses |
| Segment | Layer 4 data unit — includes port numbers |
| Protocol | A set of rules two machines agree to follow when communicating |
| Encapsulation | Adding headers as data moves down the OSI layers |

## Day 02 — TCP/IP & Packet Tracing

### What is TCP/IP?

TCP/IP is the actual protocol suite the internet runs on. While OSI is a theoretical model with 7 layers, TCP/IP is the practical implementation with 4 layers:

| TCP/IP Layer | Equivalent OSI Layers | Key Protocols |
|---|---|---|
| Application | 5, 6, 7 | HTTP, DNS, FTP, SSH, SMTP |
| Transport | 4 | TCP, UDP |
| Internet | 3 | IP, ICMP, ARP |
| Network Access | 1, 2 | Ethernet, Wi-Fi |

---

### TCP — Transmission Control Protocol

TCP is **reliable** and **connection-oriented**. Before sending any data, TCP establishes a connection using the **3-way handshake**.

#### The 3-Way Handshake

```
Client                          Server
  |                               |
  |-------- SYN ----------------->|   "I want to connect, my sequence number is X"
  |                               |
  |<------- SYN-ACK --------------|   "OK, my sequence number is Y, acknowledging X+1"
  |                               |
  |-------- ACK ----------------->|   "Acknowledged Y+1, connection established"
  |                               |
  |====== DATA FLOWS BOTH WAYS ===|
```

- **SYN** = Synchronise — client initiates connection
- **SYN-ACK** = Server acknowledges and sends its own synchronisation
- **ACK** = Client acknowledges — connection is now open

**Why this matters for pentesting:** A SYN flood attack sends thousands of SYN packets but never completes the handshake, exhausting the server's connection table. A half-open scan (Nmap -sS) sends SYN, waits for SYN-ACK to confirm a port is open, then sends RST instead of ACK — leaving no log entry because the connection was never completed.

#### TCP Connection Termination — 4-Way Handshake

```
Client ----FIN----> Server    "I'm done sending"
Client <---ACK----- Server    "OK, noted"
Client <---FIN----- Server    "I'm done too"
Client ----ACK----> Server    "OK, bye"
```

#### TCP vs UDP

| Feature | TCP | UDP |
|---|---|---|
| Connection | Connection-oriented (handshake) | Connectionless |
| Reliability | Guaranteed delivery, retransmits | No guarantee |
| Order | Packets arrive in order | May arrive out of order |
| Speed | Slower (overhead) | Faster (no overhead) |
| Use cases | HTTP, SSH, FTP, email | DNS, video streaming, VoIP, games |

---

### IP Addressing

Every device on a network has an IP address — a logical address that identifies it at Layer 3.

#### IPv4 Address Structure
- 32 bits, written as 4 octets: `192.168.1.100`
- Each octet is 0–255
- Two parts: **network portion** and **host portion** (determined by subnet mask)

#### Private IP Ranges (RFC 1918)
These ranges are not routable on the public internet — they're used inside private networks:

| Range | CIDR | Common use |
|---|---|---|
| 10.0.0.0 – 10.255.255.255 | 10.0.0.0/8 | Large enterprise networks |
| 172.16.0.0 – 172.31.255.255 | 172.16.0.0/12 | Medium networks |
| 192.168.0.0 – 192.168.255.255 | 192.168.0.0/16 | Home/small office networks |

**Why this matters:** When you're doing internal pentests or working inside VMs, you'll always see these ranges. `192.168.x.x` means you're on a local network. Knowing the ranges immediately tells you the network's scale and whether you can reach the internet directly.

#### Special IP Addresses
- `127.0.0.1` — loopback (localhost) — your own machine talking to itself
- `0.0.0.0` — means "all interfaces" when binding a server
- `255.255.255.255` — broadcast — send to all devices on the network
- `x.x.x.1` — usually the default gateway (your router)

---

### Ports

Ports are numbers (0–65535) that direct traffic to the right application on a machine. IP gets data to the right machine, ports get it to the right program.

#### Well-Known Ports (0–1023)
| Port | Protocol | Service |
|---|---|---|
| 21 | TCP | FTP (File Transfer) |
| 22 | TCP | SSH (Secure Shell) |
| 23 | TCP | Telnet (unencrypted remote access) |
| 25 | TCP | SMTP (email sending) |
| 53 | TCP/UDP | DNS (domain name resolution) |
| 80 | TCP | HTTP (web) |
| 443 | TCP | HTTPS (encrypted web) |
| 445 | TCP | SMB (Windows file sharing) |
| 3306 | TCP | MySQL database |
| 3389 | TCP | RDP (Windows Remote Desktop) |
| 8080 | TCP | Alternative HTTP / web proxies |

**Memorise these.** When Nmap shows you an open port, you'll instantly know what service is probably running there.

---

### Wireshark — Reading Real Traffic

Wireshark captures packets off your network interface and lets you read them layer by layer.

#### Key Wireshark Filters
```
tcp                         — show only TCP packets
udp                         — show only UDP packets
http                        — show only HTTP traffic
dns                         — show only DNS queries and responses
ip.addr == 192.168.1.1      — show only traffic to/from this IP
ip.src == 192.168.1.1       — show only traffic FROM this IP
ip.dst == 192.168.1.1       — show only traffic TO this IP
tcp.port == 80              — show only traffic on port 80
tcp.flags.syn == 1          — show only SYN packets (connection attempts)
```

#### How to Follow a TCP Stream
Right-click any packet → Analyze → Follow → TCP Stream

This reassembles the full conversation between client and server into readable text. For unencrypted HTTP traffic you'll see the raw request and response. This is how you extract credentials from captured traffic in CTFs.

---

### Tracing a Packet — Full Journey

When you type `http://example.com` in your browser:

```
1. DNS Query
   Your machine → DNS server: "What's the IP for example.com?"
   DNS server → Your machine: "It's 93.184.216.34"

2. TCP Handshake (to port 80)
   Your machine → example.com: SYN
   example.com → Your machine: SYN-ACK
   Your machine → example.com: ACK

3. HTTP Request
   Your machine → example.com: GET / HTTP/1.1

4. HTTP Response
   example.com → Your machine: HTTP/1.1 200 OK + HTML content

5. TCP Teardown
   FIN → ACK → FIN → ACK
```

Every one of these steps is visible in Wireshark as separate packets.

## Day 03 — DNS, DHCP & ARP

### DNS — Domain Name System

DNS translates human-readable domain names into IP addresses. It's often called "the phonebook of the internet."

#### DNS Resolution Chain

```
Your Browser
     |
     v
1. Check local cache (already looked this up recently?)
     |
     v
2. Check /etc/hosts (Windows: C:\Windows\System32\drivers\etc\hosts)
     |
     v
3. Ask Recursive Resolver (usually your ISP or 8.8.8.8)
     |
     v
4. Recursive Resolver asks Root Nameserver → "who handles .com?"
     |
     v
5. Root Nameserver → "ask the .com TLD nameserver"
     |
     v
6. TLD Nameserver → "ask example.com's authoritative nameserver"
     |
     v
7. Authoritative Nameserver → "example.com is 93.184.216.34"
     |
     v
8. Answer returned to your browser, cached for future use
```

#### DNS Record Types
| Record | Purpose | Example |
|---|---|---|
| A | Maps domain to IPv4 address | example.com → 93.184.216.34 |
| AAAA | Maps domain to IPv6 address | example.com → 2606:2800::1 |
| MX | Mail server for the domain | example.com → mail.example.com |
| CNAME | Alias — points to another domain | www.example.com → example.com |
| TXT | Text data — used for SPF, verification | "v=spf1 include:..." |
| NS | Nameserver records — who handles DNS for this domain | ns1.example.com |
| PTR | Reverse DNS — IP to domain name | 93.184.216.34 → example.com |

**Why this matters for pentesting:** DNS records expose infrastructure. MX records reveal mail servers. NS records reveal hosting providers. TXT records sometimes leak internal information. Zone transfers (AXFR) can dump the entire DNS database if misconfigured — giving you every subdomain at once.

#### DNS Tools on Windows

```cmd
nslookup example.com                    — basic lookup
nslookup -type=MX example.com          — query MX records
nslookup -type=TXT example.com         — query TXT records
nslookup example.com 8.8.8.8           — use Google's DNS server specifically
```

```powershell
Resolve-DnsName example.com            — PowerShell equivalent
Resolve-DnsName example.com -Type MX   — query specific record type
```

#### DNS and Security
- **DNS poisoning** — feeding false DNS records to a resolver so victims get sent to a malicious server
- **DNS tunnelling** — encoding data inside DNS queries to exfiltrate information past firewalls
- **Subdomain enumeration** — guessing or bruteforcing subdomains (admin.example.com, dev.example.com) to find hidden services
- **Zone transfer** — if misconfigured, `nslookup -type=AXFR example.com ns1.example.com` dumps all DNS records

### DHCP — Dynamic Host Configuration Protocol

DHCP automatically assigns IP addresses to devices when they join a network. Without DHCP you'd have to manually configure every device's IP, subnet mask, gateway, and DNS server.

#### DHCP Process (DORA)
```
Client                          DHCP Server
  |                               |
  |------ DISCOVER (broadcast) -->|   "I need an IP address, anyone there?"
  |                               |
  |<----- OFFER ------------------|   "I can offer you 192.168.1.100"
  |                               |
  |------ REQUEST --------------->|   "Yes, I'll take 192.168.1.100"
  |                               |
  |<----- ACK --------------------|   "It's yours for 24 hours (lease time)"
```

**Why this matters for pentesting:** A rogue DHCP server attack puts your own DHCP server on the network to hand out IP addresses with your machine as the gateway — all traffic flows through you. This is how DHCP-based man-in-the-middle attacks work.

---

### ARP — Address Resolution Protocol

ARP resolves IP addresses to MAC addresses within a local network. Layer 3 knows the destination IP, but Layer 2 needs the MAC address to actually deliver the frame.

#### ARP Process
```
Device A wants to send to 192.168.1.50 but doesn't know its MAC

Device A broadcasts: "Who has 192.168.1.50? Tell 192.168.1.1"
        (sent to FF:FF:FF:FF:FF:FF — the broadcast MAC address)

Device B (192.168.1.50) replies: "I have 192.168.1.50, my MAC is AA:BB:CC:DD:EE:FF"

Device A stores this in its ARP cache and sends the frame directly to AA:BB:CC:DD:EE:FF
```

#### ARP Cache
```cmd
arp -a          — view your ARP cache (IP to MAC mappings)
```

#### ARP Spoofing — Why ARP is Dangerous
ARP has no authentication. Any device can send a fake ARP reply saying "I am 192.168.1.1 (the router), my MAC is MY_MAC." Every device that receives this updates its ARP cache and starts sending traffic meant for the router to you instead — this is a **man-in-the-middle attack** at Layer 2.

This is one of the most common local network attacks and tools like `arpspoof` and Bettercap automate it completely.

## Day 04 — HTTP, HTTPS & TLS

### HTTP — HyperText Transfer Protocol

HTTP is the protocol your browser uses to request web pages. It operates at Layer 7 (Application) and is built on top of TCP at Layer 4.

#### HTTP Request Structure
```
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html,application/xhtml+xml
Connection: keep-alive

[empty line signals end of headers]
[optional request body for POST requests]
```

#### HTTP Methods
| Method | Purpose | Has Body? |
|---|---|---|
| GET | Retrieve a resource | No |
| POST | Submit data to the server | Yes |
| PUT | Replace a resource entirely | Yes |
| PATCH | Partially update a resource | Yes |
| DELETE | Remove a resource | No |
| HEAD | Like GET but returns only headers | No |
| OPTIONS | Ask what methods the server supports | No |

**Why this matters for pentesting:** Most web vulnerabilities involve manipulating HTTP requests. SQL injection goes in GET parameters or POST bodies. File uploads use POST. Understanding the method and where parameters live tells you where to inject.

#### HTTP Response Structure
```
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1256
Server: Apache/2.4.41
Date: Mon, 01 Jan 2024 12:00:00 GMT

[empty line]
[response body — the HTML content]
```

#### HTTP Status Codes
| Code | Meaning | Pentesting relevance |
|---|---|---|
| 200 | OK — request succeeded | Normal response |
| 301/302 | Redirect | Follow the redirect |
| 400 | Bad Request — malformed syntax | Your injection may have broken something |
| 401 | Unauthorized — authentication required | Login wall |
| 403 | Forbidden — authenticated but no permission | IDOR potential |
| 404 | Not Found | Directory busting shows this for non-existent paths |
| 500 | Internal Server Error | Your injection may have caused an error — potential SQL injection indicator |
| 503 | Service Unavailable | Server overloaded or down |

**Key insight:** A 500 error in response to an unusual input often means you've broken something in a way that could be exploited. A 403 instead of 404 means the resource exists but you can't access it — worth investigating further.

### HTTPS — HTTP Secure

HTTPS is HTTP with TLS encryption layered on top. The data is identical — the difference is that it's encrypted in transit so nobody between you and the server can read it.

#### The TLS Handshake

```
Client                                  Server
  |                                       |
  |-- ClientHello ----------------------->|
  |   (TLS version, cipher suites,        |
  |    random number)                     |
  |                                       |
  |<-- ServerHello ----------------------|
  |   (chosen cipher suite,              |
  |    server's random number)           |
  |                                       |
  |<-- Certificate ---------------------|
  |   (server's public key + identity    |
  |    signed by a trusted CA)           |
  |                                       |
  |   [Client verifies certificate]       |
  |                                       |
  |-- ClientKeyExchange --------------->|
  |   (pre-master secret encrypted       |
  |    with server's public key)         |
  |                                       |
  |   [Both sides derive session keys]    |
  |                                       |
  |-- ChangeCipherSpec + Finished ------>|
  |<-- ChangeCipherSpec + Finished ------|
  |                                       |
  |======= ENCRYPTED DATA FLOWS ========|
```

#### Key Cryptography Concepts

**Asymmetric encryption (used during handshake):**
- Public key encrypts, private key decrypts
- The server's public key is in its certificate — anyone can see it
- Only the server's private key can decrypt what's encrypted with the public key
- Used to securely exchange the session key

**Symmetric encryption (used for data transfer):**
- Same key encrypts and decrypts
- Much faster than asymmetric
- AES-256 is the standard — this encrypts your actual data
- The session key is established during the handshake and used for the rest of the connection

**Certificate Authorities (CAs):**
- Trusted third parties that vouch for a server's identity
- Your browser has a built-in list of trusted CAs
- When a server presents a certificate, your browser checks: was this signed by a CA I trust?
- Self-signed certificates trigger browser warnings because no CA vouched for them

#### Wireshark and HTTPS
You cannot read HTTPS traffic in Wireshark unless you have the server's private key or the session keys. This is by design. You'll see `TLSv1.3 Application Data` — encrypted blobs you can't read. This is why attackers use tools like Burp Suite as a proxy — Burp sits between your browser and the server, acting as a man-in-the-middle with its own certificate, decrypting traffic on both sides.

---

### Using curl for Raw HTTP Requests

`curl` lets you make HTTP requests from the command line — essential for testing APIs and web endpoints during pentests.

```cmd
curl http://example.com                             — basic GET request
curl -v http://example.com                          — verbose — shows headers
curl -I http://example.com                          — HEAD request — headers only
curl -X POST http://example.com/login -d "user=admin&pass=test"   — POST request
curl -H "Authorization: Bearer TOKEN" http://api.example.com      — add a header
curl -L http://example.com                          — follow redirects
curl -o output.html http://example.com              — save response to file
curl -k https://example.com                         — ignore SSL cert errors
```

**Key insight:** The `-v` flag is your friend. It shows you the full request and response headers, which tells you exactly what the server is and what it accepts. During a pentest, `curl -v` is often the first thing you run against a target URL.

---

### HTTP vs HTTPS in Wireshark

| Traffic Type | What you see in Wireshark |
|---|---|
| HTTP | Full request and response in plaintext — readable |
| HTTPS (TLS 1.2 and below) | Handshake visible, data encrypted |
| HTTPS (TLS 1.3) | More of the handshake is encrypted, harder to analyse |
